### 4.1 Требует изменения схемы Prisma (миграция)

| P1-2 (production) | `StockMovement`: `unitCost Decimal? @db.Decimal(12,4)`, `totalCost Decimal? @db.Decimal(12,4)`, `productionJobId String?` + relation на `ProductionJob` — для финансового снапшота списаний. | Средний |

### 4.2 Требует решения владельца (политика/инфраструктура, без правки схемы либо с ней)

- **P0-5-hmac-nonce-replay (sync)** — привязать HMAC к хэшу тела запроса + одноразовый nonce-store. Требует байт-стабильной сериализации payload на обоих узлах и персистентного хранилища nonce. **Наивысший остаточный риск (см. §6).**
- **P0-5-secret-rotation (sync)** — процедура ротации с dual-secret overlap-окном и сроком годности.
- **P1-8 TTL (auth)** — короткий TTL для привилегированных сессий (config `JWT_EXPIRES_IN` / per-role).
- **P1-9e (audit)** — логирование неуспешной аутентификации (401) в `JwtAuthGuard` или global `ExceptionFilter`.
- **P2-5 (public)** — rate-limit (`@nestjs/throttler` + `ThrottlerModule` в app.module; multi-node, per-process счётчики неэффективны), OTP (store + SMS/Telegram), CAPTCHA/PoW для `POST /public/orders` и password-reset.
- **P1-1 (cash)** — опц. новый код права `cash.manage` в `auth/permissions.ts` + seed, если нужен manager-override (закрытие/правка чужих смен). Сейчас — строго own-userId.
- **P1-5 (search)** — опц. выделенное право `search.view`, если глобальный поиск должен быть доступен ролям без `orders.view` (сейчас входной гейт — `orders.view`).
- **P2-10 (web)** — политика глобального 403-UX (баннер/редирект). Сейчас 403 намеренно не централизован, чтобы не сломать silent-403 (например, кассир без `orders.view` в POS).
- **Sync opt-in флаги** — включение `SYNC_INCLUDE_SENSITIVE=1` вновь открывает blind-writes; отключение legacy shared-secret (`x-sync-secret`) — операционное решение.

## 6. Наивысший остаточный риск

**P0-5-hmac-nonce-replay (sync).** Это единственный незакрытый пункт уровня P0. HMAC подписывает только `${node}.${timestamp}` и не привязан к телу запроса; nonce отсутствует. Перехваченная подпись может быть переиграна в окне 5 минут — потенциально с другим телом push, что даёт запись в БД от имени доверенного узла. Требует привязки подписи к хэшу тела и одноразового nonce-store (схема/Redis) — оба изменения риск-нагруженные для живого sync, поэтому вынесены на решение владельца, а не «дожаты» вслепую.

## 7. Рекомендованные следующие шаги и порядок коммитов

Изменения затрагивают несколько модулей; рекомендуется серия небольших тематических коммитов (легче ревью/откат), а не один большой.

**Порядок коммитов (по готовности к мержу):**

1. **`git add printpro-web/src/app/(panel)/reports/sections.tsx`** — файл untracked, иначе правки P2-8 не попадут в коммит/деплой. Сделать **первым делом**.
2. `fix(api): sync — peer-identity аудит push/pull (P0-5)` — `sync.controller.ts`, `sync.service.ts`.
3. `fix(api): roles — запрет изменения системных ролей (P0-4)` — `roles.service.ts`.
4. `fix(api): cash/payroll — обязательная открытая смена и own-userId ownership (P1-1, P1-7)` — `cash.service.ts`, `payroll.service.ts`.
5. `fix(api): orders — централизация смены статуса через транзакционные хелперы (P1-10)` — `orders.service.ts`.
6. `fix(api): search — per-entity проверка прав по роли (P1-5)` — `search.controller.ts`, `search.service.ts`.
7. `fix(api): auth/audit/production — isActive в me(), покрытие public/sync аудитом, userId в списании (P1-8, P1-9, P1-2)` — `auth.service.ts`, `audit.interceptor.ts`, `production.*`.

**После мержа — задачи в бэклог (в порядке приоритета):**

1. **P0-5 replay-защита** (наивысший риск) — HMAC к хэшу тела + nonce-store; отключить legacy shared-secret в проде.
2. **P1-8 отзыв токенов** — миграция `User.tokenVersion` (или Session) + сравнение в guard/`me()`; политика короткого TTL для привилегий.
3. **P2-5 rate-limit/OTP/CAPTCHA** на публичных эндпоинтах.
4. **P1-2 финснапшот списаний** — миграция `StockMovement.unitCost/totalCost/productionJobId`.
5. **P1-9d/e аудит** — before/after внутри транзакций денег/склада/RBAC; логирование 401 в guard/ExceptionFilter.
6. **P0-5 ротация секретов**, **P0-7 Return.idempotencyKey** — hardening по мере необходимости.

**Замечание по процессу (Windows):** перед `prisma migrate`/`generate` останавливать dev-сервер API (держит DLL Prisma). Обновить `docs/СТАТУС-ПРОЕКТА.md` по итогам сессии.
