# Аудит бизнес-логики PrintPro — 2026-07-02

Полный прогон кода 7 независимыми аудиторами по доменам: заказы/продажи, касса/смены,
склад/товары, закупки/поставщики, клиенты/бонусы/долги, безопасность/мультитенантность,
логика POS-фронта. Все находки подтверждены чтением кода (файл:строка). Многие баги
найдены независимо несколькими аудиторами — отмечено.

Легенда: 🔴 критично · 🟠 высоко · 🟡 средне · ⚪ низко.

---

## 🔴 КРИТИЧНЫЕ (чинить в первую очередь — теряются деньги/данные, дыра в безопасности)

### К1. Оплата методом «В долг» гасит долг без денег
`printpro-api/src/orders/orders.service.ts:306-333`, DTO `order-actions.dto.ts:17,43`.
`addPayment` не блокирует `method=DEBT`, а DTO его принимает. Кассир проводит оплату
«в долг» на странице заказа или частью смешанной оплаты (`[CASH 50, DEBT 50]`) → `paid`
растёт, заказ становится PAID, долг исчезает, денег нет. Подтвердили 3 аудитора.
**Фикс:** в `addPayment` отклонять `DEBT` (BadRequest); в `payments[]` quickSale запретить DEBT-части.

### К2. Кросс-компанийный доступ (мультитенантный IDOR)
Десятки эндпоинтов берут `companyId` из query/body клиента, не из JWT, а `:id`-маршруты
не сверяют владельца. Подтверждено: `orders.controller.ts:55,68,101` (findAll/listHeld/listReturns),
`orders.service.ts:278,570,657,832,952` (addPayment/refund/createReturn/updateStatus/findOne по id),
`warehouse/stock.controller.ts` (весь), `purchasing.controller.ts:32,61`, `settings.controller.ts:14,21`,
`cash.controller.ts:46` + `cash.service.ts:125` (Z-отчёт чужой смены), `backup.controller.ts:19`,
`audit.controller.ts:15`, `roles.service.ts:15,28`.
Сотрудник компании А, подставив чужой `companyId`/`id`: читает/пишет чужие остатки, заказы,
поставщиков; **выгружает полный бэкап** чужой компании; перезаписывает чужие настройки;
получает Z-отчёт чужой кассы. Подтвердили 3 аудитора.
**Фикс:** во всех контроллерах брать `companyId` только из `@CurrentUser()`, query-параметр убрать;
в сервисах добавить `companyId` в `where` всех запросов по `id`. Часть модулей уже так делает
(clients/cash-open/payroll/users/quotes/production/tasks/promocodes) — привести остальные.

### К3. quickSale не в одной транзакции — при откате теряются бонусы и промокод
`orders.service.ts:408-535`. Скидка, `promocodes.consume` (инкремент usedCount), списание
бонусов и оплаты идут вне общей транзакции; компенсация — `refund().catch(()=>{})` (:533),
которая НЕ возвращает бонусы и не откатывает использование промокода, а ошибку глотает.
Сценарий: части смешанной оплаты не сошлись с итогом (:460) → заказ отменён, клиент навсегда
потерял списанные бонусы; при повторе — теряет снова. Если сам refund упал — товар списан,
заказ висит активным неоплаченным. Подтвердили 4 аудитора.
**Фикс:** всю пост-обработку quickSale провести в одной `$transaction`; в откате восстанавливать
`bonusPoints` и декрементить `usedCount` промокода.

### К4. Повтор оплаты после ошибки возвращает ОТМЕНЁННЫЙ заказ как успешный чек
`pos/page.tsx:509-516` + `orders.service.ts:372-377,531-535`. При ошибке после создания
заказа бэк делает `refund()` (CANCELLED), но `idempotencyKey` не очищает. Кассир исправляет
сумму, жмёт снова → бэк по тому же ключу возвращает отменённый заказ через `findOne` как готовый.
Фронт показывает чек, деньги не проведены, товар не списан.
**Фикс:** при откате очищать `idempotencyKey` (или искать по ключу только `status != CANCELLED`);
на фронте ротировать ключ при 4xx, сохранять только при сетевых ошибках.

### К5. Возврат не сторнирует бонусы — «фарм» бонусов
`orders.service.ts:568-647` (refund), `:652-781` (createReturn). Не отменяются начисленные
1% и не возвращаются списанные бонусы. Клиент покупает за 1000 (получает 10 бонусов), делает
полный возврат — деньги вернули, бонусы остались; цикл купил-вернул бесплатно копит бонусы.
Подтвердили 2 аудитора.
**Фикс:** в refund/createReturn сторнировать начисленные и возвращать списанные бонусы
пропорционально возвращённой сумме.

## 🟠 ВЫСОКИЕ

### В1. Повторный частичный возврат раздувает склад и returnedTotal
`orders.service.ts:667-714`. Количество ограничено только `min(ri.quantity, oi.quantity)` за
вызов, ранее возвращённое не учитывается (документы Return не суммируются, статус заказа не
проверяется). Двойной возврат 5 шт по позиции в 5 шт → на склад 10 шт, `returnedTotal > total`.
Подтвердили 3 аудитора. **Фикс:** считать возвращённое по `orderItemId` из прошлых Return,
ограничивать остатком; запрещать возврат по CANCELLED.

### В2. Полный refund после частичного возврата приходует товар дважды
`orders.service.ts:597-636`. Возвращает полное `it.quantity`, игнорируя прошлые RETURN и
признак списания (флага `decrementStock` в модели Order нет). Частичный возврат 2 из 5, затем
«Возврат заказа» → на склад придёт ещё 5 (итого 7 из 5). **Фикс:** вычитать ранее возвращённые
количества; хранить признак списания на заказе.

### В3. Гонки без блокировки строки (read-then-write вне транзакции)
Единый класс багов — двойной клик/параллельные запросы:
- Списание бонусов в минус — `orders.service.ts:425-443`.
- Переплата заказа двойным кликом — `orders.service.ts:278-333`.
- Двойное открытие смены — `cash.service.ts:19-34` (нет unique-индекса «одна открытая смена»).
- Двойное списание материалов производства — `production.service.ts:86-95`.
- Двойная оплата долга поставщику — `purchasing.service.ts:32-49`.
**Фикс:** атомарный условный `updateMany({ where: { ..., поле >= n }, data: { decrement } })`
с проверкой `count`, как уже сделано для склада и промокодов; для смены — частичный уникальный
индекс `WHERE closedAt IS NULL`.

### В4. Возврат всегда как наличный расход, даже при оплате картой/QR/переводом
`orders.service.ts:583-595,729-741`. `CashMovement OUT` создаётся без учёта метода исходных
платежей. Заказ оплачен картой, возврат → `expectedCash` занижен, у кассира ложная «недостача».
Подтвердили 2 аудитора. **Фикс:** наличный OUT только для CASH-части; безналичные возвраты —
отдельно (отрицательный Payment соответствующего метода).

### В5. Смена статуса на CANCELLED стирает долг без возврата денег/товара
`orders.service.ts:826-841,979-987`. `updateStatus` допускает любой переход; список долгов
фильтрует `status != CANCELLED`. `PATCH /orders/:id/status {CANCELLED}` по долговому заказу на
500 → долг исчез, товар списан, деньги (если были) не возвращены. **Фикс:** переход в CANCELLED
только через `refund()`; добавить матрицу допустимых переходов статусов.

### В6. Расхождение сумм фронт↔бэк (бонусы и промокод) — на экране одна сумма, в чеке другая
`pos/page.tsx:224-227,412-419` vs `orders.service.ts:410-436`. (а) Фронт шлёт сырой `useBonus`,
а не обрезанный `bonusApplied`, и не знает баланс клиента → для нового клиента «бонус применён»,
а баллов 0. (б) База 30% различается (фронт — после скидки/промо, бэк — от полного subtotal).
(в) База промокода различается (фронт `afterDisc`, бэк полный `total`). Второй экран покупателя
показывает сумму, которая не совпадёт с чеком. **Фикс:** слать обрезанные значения, подтягивать
`bonusPoints` при вводе телефона, унифицировать базы расчёта на обеих сторонах.

### В7. Безопасность входа и загрузок
- `pos-login` без rate-limit — брутфорс 4-значного PIN (`auth.service.ts:54-70`).
- Uploads без проверки MIME, раздача статики с origin API (`uploads.controller.ts:20`, `main.ts:30`)
  → загрузка `.html`/`.svg` с JS = stored XSS.
- `/public/upload` вообще без авторизации (`public.controller.ts:75`).
**Фикс:** @nestjs/throttler на login/pos-login; whitelist MIME (только изображения), отдавать с
`Content-Disposition: attachment`; закрыть публичную загрузку.

## 🟡 СРЕДНИЕ

- **С1. Дубликаты клиентов** — телефон не нормализуется, на `(companyId, phone)` только индекс,
  не unique; findFirst→create даёт гонку. `clients.service.ts:21-34`, `schema.prisma:316`.
- **С2. Заявку на закупку можно «принять» многократно** — у `PurchaseRequest` нет статуса,
  приёмка с ней не связана → двойное оприходование остатков и долга. `schema.prisma:679`, `purchasing.service.ts:93`.
- **С3. Оплата долга поставщику не гасит приёмки** — `paySupplierDebt` уменьшает `supplier.debt`,
  но `StockReceipt.paymentStatus` навсегда «В долг · просрочено». `purchasing.service.ts:38-64`.
- **С4. Дубликаты штрихкодов** — нет unique-констрейнта + check-then-act; `generateBarcode` не
  резервирует код. `schema.prisma:536`, `products.service.ts:17-48,266-294`.
- **С5. POS-фильтр по родительской категории не показывает товары подкатегорий** — строгое
  `i.categoryId === catFilter`. `pos/page.tsx:180`, `_pos.tsx:781,1228,1665`, `_pos-prime.tsx:688`.
- **С6. PATCH товара затирает категорию/единицу** — `categoryId: dto.categoryId ?? null` при
  частичном апдейте. `products.service.ts:392-393`.
- **С7. Денежная операция без открытой смены проходит молча (shiftId=null)** — не попадает ни в
  один Z-отчёт (оплаты, возвраты, оплата поставщику, зарплата). `orders.service.ts:298-304`,
  `purchasing.service.ts:52`, `payroll.service.ts:252`.
- **С8. Отложенные чеки теряют скидку/промокод/бонус/клиента/примечание** — `hold` шлёт только
  `label,total,items`. `pos/page.tsx:546-569`, `orders.service.ts:541-553`.
- **С9. Второй экран: «вечное» состояние + общий канал на все вкладки** — `pp_display_last`
  не сбрасывается; две вкладки кассы перебивают друг друга (покупатель видит чужую корзину).
  `customer-display.ts:11,62-86`.
- **С10. Произвольная скидка с фронта без лимита и прав** — `discount = total` обнуляет чек.
  `order-actions.dto.ts:65`, `orders.service.ts:446`.
- **С11. Удаление родительской категории** — дети всплывают наверх, товары не открепляются,
  hard delete ломает soft-delete/синхронизацию, не в транзакции. `products.service.ts:374-381`.
- **С12. Массовая инвентаризация** упрётся в 5-сек таймаут транзакции Prisma на 300+ позиций.
  `stock.service.ts:289-322`.
- **С13. Закрытие смены не атомарно, без проверки владельца, двойное закрытие.** `cash.service.ts:48-72`.
- **С14. Кредитный лимит** проверяется вне транзакции и по сумме до скидок. `orders.service.ts:116-134`.
- **С15. Сканер: при фокусе в поле скан не работает; при открытом чеке наполняет скрытую корзину.**
  `pos/page.tsx:341-362`.
- **С16. Публичные эндпоинты** раскрывают реквизиты компании и историю заказов клиента по телефону
  без подтверждения. `public.controller.ts`, `public-settings.controller.ts:11`.
- **С17. Тест-уведомления** (`email/test`, `telegram/test`) берут companyId из body без прав →
  рассылка через чужой SMTP на произвольный адрес. `notifications.controller.ts:31-49`.
- **С18. paymentStatus=DEBT теряется при частичном погашении** + DEBT-Payment на весь итог не
  сторнируется → сумма платежей по заказу = 2× итога. `orders.service.ts:321-324,505-514`.

## ⚪ НИЗКИЕ (качество/точность/UX)

- Округления на float (`toFixed(2)` теряет дирам на позициях вида 3×0.335); считать через `Prisma.Decimal`.
- `idempotencyKey` глобально уникален без companyId (`schema.prisma:805`).
- `deleteHeld`/`adjust` без проверки companyId; `adjust` принимает тип IN/RETURN но всегда списывает.
- Импорт CSV/Excel: не в транзакции, нет валидации значений, тихая потеря штрихкодов, слияние по имени.
- Z-отчёт не фильтрует soft-deleted платежи/движения (`cash.service.ts:131-137`).
- Сдача считается, но не показывается в чеке/на дисплее/термочеке; кнопка оплаты не блокируется на время запроса.
- JWT без отзыва/refresh (7д / PIN 12ч); `JWT_SECRET` не валидируется при старте; пароль от 4 символов; CORS открыт.
- EAN-13: контрольная цифра не проверяется при ручном вводе (только при генерации).
- `promocodes.consume` не перепроверяет isActive/validUntil атомарно.
- Страница «Долги» показывает все неоплаченные заказы (в т.ч. в работе), не только `paymentStatus=DEBT`.
- «Удалённый» клиент виден/редактируем по прямой ссылке (`clients.service.ts:157,249` без `deletedAt: null`).
- `DocumentCounter` без года в уникальном ключе (нумерация не сбрасывается 1 января).

## Чего не хватает (функциональность)

- Редактирование заказа после создания (позиции/суммы) с пересчётом paid/balanceDue.
- Валидация переходов статусов (state machine).
- Модель `ClientDebt` объявлена в схеме, но нигде не используется — долги живут только в
  `Order.balanceDue`; нет журнала погашений и истории частичных оплат.
- Журнал бонусов (`BonusTransaction`) — баланс меняется голыми increment/decrement без истории.
- Возврат по методу оплаты (сейчас всё — наличный расход).
- Отмена/сторно и редактирование приёмки; статусы заявки на закупку; история оплат поставщику (UI).
- Документ инвентаризации и накладная перемещения между филиалами (со статусом «в пути»).
- Резервирование материалов под заказ; себестоимость на момент продажи (FIFO/средневзвешенная).
- Серверный поиск по штрихкоду (`GET /products/by-barcode`) вместо загрузки всего каталога.
- Нормализация телефона; удаление/слияние клиентов; персональная скидка клиента при продаже.
- Rate-limiting (login/pos-login/public/uploads); отзыв токенов; валидация MIME загрузок.
- Фиксация недостачи/излишка при закрытии смены; перенос остатка между сменами; запрет продаж при закрытой кассе.

## Что реализовано хорошо (для контекста)

- Все пути списания склада защищены условным `updateMany ... quantity >= n` внутри `$transaction`
  — перерасход последней штуки двумя кассирами невозможен.
- Нумерация документов race-safe через `INSERT ... ON CONFLICT ... RETURNING` (`next-number.ts`).
- Алгоритм контрольной цифры EAN-13 при генерации корректен.
- Мягко удалённые товары исключены из остатков, каталога и lowStock.
- Публичный `/settings/ui` отдаёт данные по whitelist — SMTP-пароль и секреты не утекают.
- Модули clients/cash/payroll/users/quotes/production/tasks/promocodes корректно берут companyId из JWT.
- Web-страница кассы согласована с API по expectedCash/totalRevenue (без пересчёта на клиенте).
