## 🔴 КРИТИЧНО — безопасность

### К1. Услуги и категории услуг — БЕЗ авторизации (проверено)
`printpro-api/src/services/services.controller.ts` и `service-categories.controller.ts` — нет `@UseGuards`. Глобального guard'а в проекте нет (проверено: `APP_GUARD`/`useGlobalGuards` отсутствуют).
**Любой человек без входа** может читать, создавать, менять и удалять услуги, цены, категории **любой компании** (`companyId` берётся из query/body). Прайс-лист полностью открыт наружу.
→ Добавить `@UseGuards(JwtAuthGuard, PermissionsGuard)` + брать `companyId` из `@CurrentUser()`.

### К2. Отчёты — `companyId` из query, а не из токена (проверено)
`reports.controller.ts` — все 20 эндпоинтов берут `@Query('companyId')`. Guard есть (нужен вход + право `reports.view`), но авторизованный пользователь одной компании может подставить чужой `companyId` в URL и увидеть **выручку, прибыль, долги, зарплаты чужой типографии** (кросс-тенант IDOR).
→ Брать `companyId` из `@CurrentUser().companyId`, query-параметр игнорировать.

### К3. Склад — операции без проверки принадлежности компании
- `products.controller.ts`: `removeUnit`, `updateUnit`, `updateCategory`, `addBarcodeAlias`, `removeBarcodeAlias` — не берут `@CurrentUser()`, нет проверки `companyId`. Зная UUID, пользователь компании A правит справочники/штрихкоды компании B.
- Мутации `Stock` (во всех сервисах) идут по ключу `productId+branchId` без проверки компании.
→ Прокинуть `companyId` из токена во все эти методы, добавить проверку владельца.

## 🔴 КРИТИЧНО — деньги и данные

### К4. Остаток склада уходит в МИНУС при авто-списании материалов
`production.service.ts:134-142` — `stock.upsert` с `decrement`/`create: quantity: -qty`. Это **единственное** место, где нет условного списания. Все остальные операции (продажа, перемещение, списание, инвентаризация) защищены атомарным `updateMany WHERE quantity >= N`. При завершении производства остаток материала может стать отрицательным → ломаются отчёты по складу и оповещения о минимальном остатке.
→ Переделать на атомарное условное списание (как в `orders.service.ts:216-267`), либо осознанно разрешить минус с явным флагом.

### К5. Скидка клиента (`Client.discount`) НЕ применяется на кассе
В схеме у клиента есть поле `discount` (%), но ни `quickSale`, ни `create` заказа, ни фронт кассы его не читают. По ТЗ (2.3, 2.9) персональная скидка клиента — обязательная функция POS. Сейчас работает только ручная скидка и промокод.
→ При выборе клиента подставлять его скидку в чек (и на бэке применять при расчёте).

### К6. Сводный отчёт и долги включают отменённые заказы
- `reports.service.ts:19-23` (`summary`) — нет фильтра `status != CANCELLED` → `billed`, `ordersCount`, `balanceDue` завышены (в `profit()` фильтр есть, в `summary()` — нет).
- `/reports/debts` (стр. 299) не исключает отменённые и не отдаёт `dueDate`/`overdue`, а `/orders/debts` — исключает. Две страницы показывают **разные суммы долга**.
→ Единый фильтр отменённых; свести обе страницы долга к одному источнику.

### К7. Прибыль и средний чек считаются некорректно
- `profit()` берёт выручку из суммы `items.lineTotal`, а `summary()` — из `order.total`. При скидке на заказ отчёт прибыли завышает выручку относительно сводки.
- `avgCheck = billed / ordersCount` — по валовой сумме до возвратов; при возвратах средний чек завышен (надо по `net`).

## 🟠 ВАЖНО — касса / чек / квитанция

### В1. Z-отчёт не показывается и не печатается при закрытии смены
Бэкенд умеет отдавать отчёт закрытой смены (`GET /cash/shifts/:id/report`), но фронт после закрытия перегружает только текущую смену (она становится `null`) → итоговый Z-отчёт «уходит в никуда». X-отчёт (промежуточный) работает.

### В2. Нет истории смен в интерфейсе
`GET /cash/shifts` (100 последних) работает, но фронт его нигде не вызывает. Кассир не видит прошлые смены и их отчёты.

### В3. Возвраты недоступны из кассы
Бэкенд полностью готов (`POST /:id/refund`, `POST /:id/return`), но на экране POS нет кнопки «Возврат» (поиск чека → возврат). Возврат можно оформить только со страницы «Заказы».

### В4. «Сохранить заказ» (скин Shop) фактически проводит оплату
`_pos.tsx:643-650` — кнопка «Сохранить заказ» вызывает `pay()` (quickSale + оплата + списание) вместо создания неоплаченного заказа на печать/ремонт.

### В5. Горячие клавиши F2/F3/F5/F7/F9/F10/F11 — только надписи
На кнопках есть метки клавиш, но обработчика `keydown` для них нет (единственный слушатель ловит только сканер ШК). Кассир не может пользоваться обещанными сокращениями.

### В6. «Быстрый товар» (скин Shop) — заглушка
`_pos.tsx:926-930` — плитка «Добавить без карточки» это статичный `div` без `onClick`.

### В7. Печать чека через `window.print()` печатает всю страницу
Нужен `@media print`, скрывающий кассу за модалкой (классы `no-print`/`receipt-print` есть, но глобальный CSS не подтверждён). Отдельной PDF-квитанции для отправки клиенту нет. QR/QR-оплата как отдельный способ — отсутствует (есть только «Перевод»).

## 🟠 ВАЖНО — заказы и производство

### В8. Нет валидации переходов статуса заказа
`orders.service.ts:1034-1089` (`updateStatus`) принимает любой статус (кроме CANCELLED) без матрицы переходов. Можно из «Принят» сразу в «Выдан», минуя дизайн/производство, или откатить «Выдан» → «Принят». Фронт показывает все 10 статусов в выпадающем списке.

### В9. Массовая смена статуса глотает ошибку
`orders/page.tsx:269-280` — `bulkChangeStatus` показывает и CANCELLED; API вернёт ошибку, но `.catch(()=>{})` её проглатывает → «операция завершена», хотя часть заказов не обновилась.

### В10. Нельзя редактировать и удалять заказ
В контроллере нет `@Patch(':id')`/`@Delete(':id')`. После создания нельзя поправить менеджера/дизайнера/срок/примечание — только статус и срок долга. Ошибочный заказ не удалить.

### В11. Дизайн и производство не связаны с заказом автоматически
- Смена статуса макета (DesignProof) не меняет статус заказа (IN_DESIGN / DESIGN_APPROVED) — всё вручную.
- При переводе заказа в производство `ProductionJob` не создаётся автоматически.

### В12. Форма нового заказа: нет менеджера и загрузки файла макета
`orders/new/page.tsx` — нет поля «ответственный менеджер» (`assignedUserId` поддержан на бэке, но не отправляется) и нет загрузки файла макета (ТЗ 2.4). Нет отдельной страницы-карточки заказа (только боковая панель).

### В13. Производство: «Продолжить» с паузы всегда ставит PRINTING
`production/page.tsx:299-306` — не помнит этап до паузы (CUTTING/BINDING вернутся на PRINTING). `completedAt` обнуляется при возврате из COMPLETED в REWORK (`production.service.ts:80`). Нет валидации переходов в API. Списание идёт типом `WRITE_OFF` (строка-литерал) без `beforeQty/afterQty`.

## 🟠 ВАЖНО — склад (функционал)

### В14. Журнал списаний не выводится на фронте
`GET /api/stock/write-offs` работает, но вкладка «Списания» показывает только форму — таблицы истории нет.

### В15. Нет пагинации/фильтров в движениях, списаниях, приёмках
- `listMovements` — жёстко `take: 200`, без фильтров по дате/товару/типу/филиалу.
- `listWriteOffs` — `take: 100`.
- `listReceipts`/`listRequests` (закупки) — `take: 100`, без фильтра по периоду/статусу; `listReceipts` ещё и не фильтрует `deletedAt` (мягко удалённые видны).
Для реального склада за пару месяцев данные упрутся в лимит.

### В16. Нет отмены/сторно приёмки и списания
Ошибочную приёмку (откат остатков + долга + расхода кассы) и ошибочное списание отменить нельзя — только правкой в БД.

### В17. Перемещения неотличимы в истории
Нет типа `TRANSFER` в `StockMovementType` — межфилиальное перемещение это пара OUT+IN, в журнале выглядит как продажа/приход (у фронта нет метки TRANSFER).

## 🟡 СРЕДНЕ — отчёты (пропуски по ТЗ)

Отсутствуют отчёты, прямо названные в ТЗ 2.12:
- **Расход материалов за период** (данные в `StockMovement` есть, отчёта нет).
- **Заказы по статусам** (сколько новых/в работе/готовых за период).

Экспорт: **Excel (xlsx) нет** (только CSV), **PDF только через печать браузера**. График выручки захардкожен на 14 дней и не реагирует на выбранный период (`reports/page.tsx:72`). У секции «Расходы» нет кнопки экспорта.

Нет событий: новый заказ, макет согласован, закрытие смены, ошибка синхронизации. **SMS-канала нет вообще.** Тумблеры `notifyLowStock/notifyDebts/notifyOrderReady` в настройках **не проверяются** бэкендом — уведомления генерируются всегда. `notifications.controller.ts` без `PermissionsGuard` (кассир может слать тест-сообщения от имени компании).

### С2. Бонусы зашиты в код
Начисление 1% (`orders.service.ts:364`) и лимит списания 30% (стр. 478) — константы, нет настройки в UI, нет истории бонусных операций и ручной корректировки.

### С4. `update` клиента передаёт сырой DTO в Prisma (`clients.service.ts:63`) — при глобально невключённом `whitelist` риск протаскивания лишних полей. Проверить `ValidationPipe({ whitelist: true })`.

## 🟡 СРЕДНЕ — схема БД

- **19 моделей** имеют `companyId` без FK-`@relation` к `Company` (StockMovement, Payment, CashShift, CashMovement, ProductionJob, AuditLog и др.) → БД не проверяет существование компании, нет `include: { company }`.
- **8 моделей** с `userId/createdById` без FK к `User`; **3** с `branchId` без FK к `Branch`; `WriteOff.productId` без FK к `Product`.
- **~21 отсутствующий индекс** на горячих полях: `AuditLog(companyId,createdAt)`, `StockMovement(companyId,createdAt)`, `Order(companyId,status)` и `(companyId,createdAt)`, `Payment/CashMovement(companyId,createdAt)`, `Product/Service/Supplier(companyId)` и т.д. Сейчас растущие таблицы читаются full-scan.
- **Каскады несимметричны**: удаление Order каскадит одних детей, но блокируется Restrict от Payment/Task/StockMovement.
- `orders.service.ts buildWhere` не фильтрует `deletedAt: null` (сейчас заказы не soft-удаляются, но поле есть — «мина» на будущее).
- `CashMovement.type` — `String` вместо enum. Две миграции с одинаковым timestamp `20260702120000`.
