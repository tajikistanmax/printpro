# PrintPro — резервное копирование и восстановление БД

> Данные типографии (заказы, клиенты, деньги, склад) — самое ценное. Этот документ
> описывает бэкап боевой БД и, главное, **проверку восстановления** (непроверенный
> бэкап = отсутствие бэкапа).

## 1. Облако (Render)

Есть два уровня защиты, лучше держать оба:

### 1.1. Встроенный бэкап Render (рекомендуется)
На **платном** плане Postgres (`starter`+) Render даёт автоматические бэкапы и
point-in-time restore (PITR). **Free-план бэкапов НЕ имеет и удаляется ~через 30 дней.**
→ Перед боевым запуском повысить план БД в `render.yaml` (`databases.printpro-db.plan`).

### 1.2. Независимый offsite-бэкап (GitHub Actions)
Пайплайн [`.github/workflows/db-backup.yml`](../.github/workflows/db-backup.yml) каждые
6 часов делает `pg_dump --format=custom` и кладёт в S3-совместимое хранилище
(Cloudflare R2 / Backblaze B2 / AWS S3). Независим от Render — переживёт даже удаление
проекта.

**Активация** — задать секреты репозитория (Settings → Secrets and variables → Actions):
| Секрет | Что это |
|---|---|
| `BACKUP_DATABASE_URL` | External Database URL боевой БД (Render → printpro-db → Connect) |
| `S3_ENDPOINT` | endpoint хранилища (R2: `https://<acc>.r2.cloudflarestorage.com`) |
| `S3_BUCKET` | имя бакета (напр. `printpro-backups`) |
| `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` | ключи доступа к бакету |
| `AWS_DEFAULT_REGION` | регион (для R2 — `auto`) |

**Ротация:** настроить lifecycle-правило на бакете (удалять объекты старше 30 дней) —
проще и надёжнее, чем чистить из workflow.

## 2. Восстановление (restore)

Из custom-дампа:
```bash
# Полное восстановление в чистую БД:
pg_restore --no-owner --no-privileges --clean --if-exists \
  --dbname="postgresql://user:pass@host:5432/dbname" printpro-YYYYMMDDThhmmssZ.dump
```
Выборочно (одна таблица): `pg_restore -t "Order" ...`.

## 3. Еженедельная проверка восстановления (ОБЯЗАТЕЛЬНО)

Раз в неделю восстановить последний бэкап в **одноразовую** БД и убедиться, что данные на
месте (миграции применены, `SELECT count(*)` по ключевым таблицам не пустой):
1. Создать временную БД (локально или отдельный Render Postgres).
2. `pg_restore` последнего дампа в неё.
3. Проверить: `SELECT count(*) FROM "Order"; SELECT count(*) FROM "Client";` и вход `admin`.
4. Удалить временную БД.

Без этого шага бэкапы могут годами быть «зелёными», но битыми — обнаружится только в аварии.

## 4. Коробка (Electron)

Встроенный Postgres коробки бэкапится локально через `pg_dump` (бинарь идёт с
`embedded-postgres`) в `userData/backups` с ротацией — заготовка в
[`electron/backup.js`](../electron/backup.js). Дополнительно владельцу видна папка
`Documents/PrintPro-Backups`. Плюс — при включённой облачной синхронизации главный ПК
шлёт данные в облако (но **синхронизация ≠ бэкап**: часть чувствительных моделей не
синкается, и это зеркало, а не история версий).

## 5. Что НЕ является бэкапом
- **Синхронизация коробка↔облако** — это репликация текущего состояния, а не история.
  Удаление/порча данных разъедется на обе стороны.
- **JSON-экспорт компании** (`/backup/export`) — «человеческий» снимок для миграции между
  версиями схемы/поддержки, но не полноценный консистентный бэкап всей БД.
