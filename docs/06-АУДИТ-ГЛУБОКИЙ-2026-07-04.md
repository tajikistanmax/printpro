# PrintPro — глубокий сквозной аудит (evidence-based)

**Дата:** 2026-07-04 · **Ветка:** `audit-fixes-2026-07-03` · **Метод:** белый ящик (чтение реального кода API + web + схемы Prisma), 6 параллельных проверяющих потоков. Живой e2e-прогон в этом проходе НЕ выполнялся (см. §8).

Каждый вывод помечен: **[П]** Подтверждено кодом · **[В]** Вероятно · **[Н]** Не подтверждено (нет данных).

## 1. EXECUTIVE SUMMARY

**Что хорошо (реально сильные места, не комплименты):**
- Денежный контур защищён профессионально: атомарная нумерация документов (`INSERT … ON CONFLICT DO UPDATE RETURNING` + `@@unique`), идемпотентность POS (`Order.idempotencyKey @unique` + обработка `P2002` + компенсирующий откат), оптимистичная блокировка оплаты (`updateMany where {id, paid: prev}`), один открытый смен на кассира через частичный уникальный индекс. Двойной клик/обрыв сети дубль-продажу не создают. **[П]**
- Склад не уходит в минус нигде: продажа/списание/перемещение/производство используют атомарный `updateMany … WHERE quantity >= n` + проверку `count`. Полный audit trail движений (`beforeQty/afterQty/type/userId`). **[П]**
- Мультитенант-изоляция на аутентифицированных маршрутах выдержана строго: `companyId` всегда из JWT, сервисы проверяют владение через `where:{id, companyId}`. Ни одного случая companyId из body/query на защищённых эндпоинтах. **[П]**
- State-machine заказа с матрицей допустимых переходов; отмена только через `refund()` (ручной перевод в CANCELLED запрещён); барьер согласования макета продублирован во всех точках входа в печать. **[П]**
- Глобальный `AuditInterceptor` пишет журнал на каждый POST/PATCH/DELETE; проведённые `Payment` неизменяемы (эндпоинтов редактирования/удаления платежа нет). **[П]**

**Что критично сломано / искажает цифры:**
- **P0 — Таймзона отчётов.** `reports.range()` снапает границы периода к календарному дню сервера (UTC на Render), тогда как бизнес в UTC+5. «Сегодня», «за смену», дневные срезы считают не тот день; касса не сходится с отчётом; на дашборде «вчера» перекрывает «сегодня». Затрагивает **каждую** периодную цифру. **[П]**
- **P0/P1 — Ретро-мутация выручки.** Возвраты/отмены привязаны к дате создания заказа + накопительному `returnedTotal`, а не к дате события → закрытый месяц пересчитывается задним числом; `net` и `collected` расходятся между месяцами. **[П]**
- **P1 — Ценообразование не считается на сервере.** Вся модель `priceTier / size / option / designSurcharge / minQuantity` хранится в БД и редакторе услуг, но **нигде не применяется** — ни на фронте, ни на бэке. Цена берётся из присланного `unitPrice`. Ядро print-MIS (автоматический расчёт тиража) не работает; нет защиты от манипуляции ценой. **[П]**
- **P1 — Приёмка нарушает изоляцию арендаторов.** `createReceipt` не проверяет принадлежность `productId/branchId/supplierId` компании → можно испортить каталог/остатки/цены чужого тенанта. **[П]**
- **P1 — Оплата отменённого заказа.** `addPayment` не проверяет `status` → на возвращённый (CANCELLED) заказ можно принять деньги: фантомная выручка за уже возвращённый товар. **[П]**
- **P1 — Кассовые операции без открытой смены** (отложенный пункт С7). Наличные принимаются с `shiftId=null` и выпадают из Z-отчёта/сверки. **[П]**

**Ключевые риски для бизнеса:** недостоверные управленческие цифры (ТЗ отчётов), скрытая утечка денег через кассу без смены и оплату отменённых заказов, ручное ценообразование вместо автоматического (потеря маржи и ошибки менеджеров), возможность порчи данных другого арендатора через приёмку.

**Можно ли запускать в прод?** Для **одного** арендатора, с обученными сотрудниками и осознанием, что отчёты нужно читать с поправкой на ТЗ — условно да (ядро денег/склада надёжно). Для **мульти-тенанта** и для доверия управленческим отчётам — **нет**, пока не закрыты P0 (таймзона, ретро-выручка) и P1 (изоляция приёмки, оплата отменённого, смена, серверное ценообразование).

## 4. BACKEND AUDIT

**Архитектура:** чистое разделение controller/service/prisma; DTO + глобальный `ValidationPipe (whitelist:true, transform:true)`; companyId перезаписывается токеном (`{...dto, companyId: user.companyId}`). Замечание: `forbidNonWhitelisted` выключен (не уязвимость), часть тел — inline-типы `@Body(){...}` без class-validator.

**Транзакционность / конкуренция:** все `$transaction` идут на дефолтном **READ COMMITTED** (без `isolationLevel`). Оптимистичные `updateMany … WHERE` корректны (Postgres перечитывает WHERE на залоченной строке). Но **`aggregate`-проверки не сериализуются** → гонка кредитного лимита (два параллельных заказа проскакивают суммарный лимит), хотя комментарий обещает защиту. `quickSale` — многошаговый поток с побочными эффектами вне единой транзакции и компенсацией `.catch(console.error)`: при краше процесса между шагами возможен рассинхрон. **[П]**

**Валидация бизнес-правил:** цена/скидка/бонус-лимит частично только на фронте (см. §5, §6). `dto.shiftId` в addPayment/addMovement не валидируется на принадлежность смене/кассиру.

**Обработка ошибок / статусы:** корректные `BadRequest/NotFound`; отмена запрещена вручную; но авто-переходы из производства не пишут `userId` в историю, а POS-выдача (`DELIVERED`) не пишет `OrderStatusHistory` вовсе.

**API-контракты:** REST по модулям, единообразно. Swagger `/api/docs` открыт без auth во всех окружениях (харднинг).

**Интеграции:** sync под `SYNC_SECRET`; email/telegram есть. `GET /sync/status` публичен (раскрывает URL облака).

Полный разбор находок бэка — в §9.

## 5. FRONTEND AUDIT

**Сильное:** двойной сабмит на **денежных** формах закрыт последовательно (`payLockRef`+`idempotencyKey`+`payBusy` в POS; `busy`+confirm в кассе/долгах/закупках/ЗП); confirm на необратимых (возвраты, отмена приёмки/списания, удаления, закрытие смены, выплата ЗП); пагинация на заказах/остатках/движениях/приёмках; empty/loading почти везде; layout прячет маршруты по правам.

**Дефекты (детали в §9):**
- POS: ручная скидка без лимита и без права (кассир может обнулить чек) — **P1**.
- Рассинхрон бонус-лимита: фронт хардкодит 30%, бэк берёт из настроек → неверная сумма/сдача при др. проценте — **P2**.
- Двойной сабмит на «неденежных» create-формах (производство, макет, товар, поставщик, сотрудник, филиал, период ЗП) — кнопки без `disabled` → дубли — **P2/P3**.
- Тихие сбои: `setStatus/uploadPhoto/saveComment/toggleCheck/saveSalary` без try/catch → падают молча; `toggleCheck` оставляет галочку визуально при ошибке (рассинхрон UI↔БД) — **P2**.
- Слабая валидация позиций нового заказа (пустой refId, quantity=0, отрицательная цена) — **P2**.
- Клиентский поиск приёмок только по загруженной странице — **P2**.
- Аванс из кассы без confirm — **P2**.
- Загрузка данных `.catch(()=>{})` без error-state (маскирует 403/сеть) — **P2/P3**.
- Мёртвый код: `BackupSection`, мёртвые кнопки/плейсхолдеры в POS-скинах, хардкод-баннер скидки; нет удаления роли — **P3**.

**Доступность/производительность:** ошибки часто только тостом/цветом; больших регрессий производительности не выявлено (пагинация есть), но `.catch` глушит сетевые ошибки.

## 6. BUSINESS PROCESS AUDIT

- **Заказ:** state-machine корректна (матрица переходов, из DELIVERED только в REWORK, из CANCELLED никуда). Расчёт цены — **не автоматизирован** (P1). Скидка занижает `total` без отдельного поля/аудита; налогов нет (уточнить у владельца — [В] by-design).
- **Макет/согласование:** версии, `approvedBy/approvedAt`, барьер печати — есть. Дефекты: при нескольких макетах достаточно одного APPROVED (P2); новая версия не сбрасывает `approvedBy` (P2); нет валидации переходов ProofStatus (можно утвердить минуя SENT — P2/P3); `proofs.length===0` → печать разрешена (нет флага «нужен макет», P3).
- **Производство:** барьер согласования во всех точках входа; `syncOrderStatus` **тянет DELIVERED назад** (P2); списание материалов необратимо при откате COMPLETED→REWORK (P2); авто-переходы без `userId` в истории (P3).
- **Склад:** минус исключён; отмена приёмки/списания — абсолютная запись остатка (lost update при гонке, P2) и не идемпотентна (двойное сторно, P2); отмена приёмки не откатывает цены товара (P2). Себестоимость = последняя закупка (не средневзвешенная) — метод, не баг (P3).
- **Касса/оплата:** операции без открытой смены → мимо Z-отчёта (P1); оплата отменённого заказа (P1); `dto.shiftId` не валидируется (порча закрытых/чужих смен, P2); закрытие чужой смены (нет userId в where, P3); реверс бонусов по текущей ставке (P3).
- **Отчёты:** таймзона (P0); ретро-возвраты (P0/P1); `salesByItem` без вычета возвратов (P1); `materialsUsage/staffPerformance/equipmentLoad` — без фильтра периода/по текущей цене (P2).
- **КП→заказ:** гонка двойной конвертации создаёт дубль (P2).

## 7. SECURITY AUDIT

**AuthN/AuthZ:** JWT + PermissionsGuard единообразно; `JwtAuthGuard` проверяет `isActive/deletedAt` на каждом запросе (деактивация мгновенна); bcrypt для пароля и PIN; данные карт не хранятся. **[П]**

**Изоляция:** на защищённых маршрутах — строгая. Дыры:
- `createReceipt` не проверяет владение product/branch/supplier — **cross-tenant запись (P1)**.
- `GET /search` без PermissionsGuard → любой сотрудник перечисляет ПДн клиентов/заказы своей компании (BFLA, P2).
- Публичный `my-orders` отдаёт `balanceDue` (долг) по одному телефону (BOLA, P2).
- Брутфорс PIN кассира (4 цифры, in-memory rate-limit по IP) — P2.
- Загрузка файлов (orders/clients/production) без фильтра типа — P3 (stored-XSS смягчён CSP `default-src 'none'` + nosniff).
- `/uploads/*` раздаётся без авторизации (обскурность по UUID, без tenant-скоупа) — P3.
- Публичные `/settings/ui` (реквизиты/ИНН) без троттлинга, `/sync/status` (URL облака), `/api/docs`, `/notifications/email/test` (спам на произвольный адрес без прав) — P3-харднинг.

Полная таблица «контроллер → guard → permissions → источник companyId» — в приложении.

## 11. ФИНАЛЬНЫЙ ВЕРДИКТ

- **Что уже нормально:** денежная дисциплина (идемпотентность, атомарная нумерация, оптимистичные блокировки), складская целостность (без минуса, полный audit trail), tenant-изоляция на защищённых маршрутах, state-machine заказа, барьер согласования, неизменяемость проведённых платежей. Это заметно выше типового уровня «внутренней CRM».
- **Что сломано:** отчётная таймзона и признание выручки/возвратов (цифры недостоверны по периодам); оплата отменённого заказа; касса без смены.
- **Чего не хватает:** серверного ценообразования (ядро print-MIS), отчётов-срезов/Z/ДДС, FK-целостности на уровне БД, автотестов, лимита скидки.
- **Где логика неверна:** ретро-возвраты, барьер «одного из многих макетов», откат производства/приёмки, гонки на `aggregate`.
- **Можно ли считать полноценной системой типографии?** Ядро — да; но до «зрелой» не хватает автоматического расчёта тиража/опций и достоверной аналитики. **Мешает зрелости:** P0 (таймзона, признание выручки), P1 (ценообразование, изоляция приёмки, оплата отменённого, смена, лимит скидки) и отсутствие тестов.

- Отчёты: сумма Payment за период == collected; net учитывает возвраты в период события; границы дня в ТЗ бизнеса; salesByItem == net.
- Безопасность: /search без права; my-orders без долга; createReceipt с чужим productId; брутфорс PIN.
