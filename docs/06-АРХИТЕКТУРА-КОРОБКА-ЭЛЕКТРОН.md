# PrintPro — Коробочная версия (Electron): архитектура и план

> **Что это за документ.** Здесь зафиксировано видение владельца: сделать платформу
> **установочной программой**, которую клиент скачивает с сайта, ставит одним кликом,
> и она работает — со **встроенной базой данных**, **офлайн и онлайн**, по **локальной сети**
> и с **облачной синхронизацией**. Чтобы не забыть и чтобы можно было собрать по шагам.
>
> **Дата фиксации:** 2026-07-05. Обновлять по мере реализации.

---

## 1. Главная цель (что хочет владелец, дословно по смыслу)

- Клиент **скачивает программу с нашего сайта → устанавливает → работает.**
  Владелец **не ходит** к каждому клиенту вручную, ничего не настраивает на месте.
- **База данных встроена** в программу. Клиент **ничего не устанавливает и не создаёт** —
  ни PostgreSQL, ни базу. Всё разворачивается само при первом запуске.
- Программа работает **офлайн** (без интернета) **и онлайн** (с облаком).
- Внутри офиса — работа **по локальной сети**: один компьютер **главный**
  (руководитель / склад), кассы подключаются к нему.
- Онлайн-облако сверху — чтобы руководитель видел всё удалённо и чтобы связать **филиалы**.
- В будущем — **мобильное приложение** (пока временно закрывается PWA — «добавить на
  главный экран», см. §8).

## 2. Ключевой принцип: **ОДИН код — меняем в одном месте**

Electron — это **не копия кода**, это **обёртка (окно)** вокруг того, что уже написано.
Меняем кассу/заказ/отчёт **один раз** → меняется **и в онлайне, и в коробке**.
Двух версий кода **нет**.

```
                ОДИН репозиторий (то, что уже есть)
        ┌──────────────────────────────────────────┐
        │  printpro-web  — формы, касса, отчёты, UI  │
        │  printpro-api  — логика, деньги, БД (Prisma)│
        └──────────────────────────────────────────┘
                 │                          │
        показывается в                показывается в
                 ▼                          ▼
        🌐 Онлайн (браузер)         💻 Electron (коробка у клиента)
```

Пример: «убери кнопку из кассы» → правим **один файл**
[`printpro-web/src/app/(panel)/cash/page.tsx`](../printpro-web/src/app/(panel)/cash/page.tsx)
→ обновилось везде. В коробке прилетит **авто-обновлением** (клиент не переустанавливает).

**Оговорка (честно):** есть маленький «скелет» Electron (3–4 служебных файла: окно,
запуск встроенной базы, авто-обновление). Пишется **один раз**, дальше почти не трогается.
Ежедневная работа (касса, склад, деньги, отчёты) его **не касается** — это общий код.

Итог: **99% работы — одно место**, меняется везде. **1% (скелет Electron)** — написан один раз.

## 3. База данных — ставится САМА, клиент ничего не устанавливает

Смысл коробки: клиент **скачал → двойной клик → готово**. Никакого PostgreSQL руками.

**Что происходит при первом запуске (всё автоматически, внутри программы):**
1. Программа распаковывает **встроенный Postgres** (лежит внутри установщика).
2. Сама создаёт базу в своей папке данных (без настроек пользователя).
3. Сама применяет миграции (`prisma migrate deploy`) → создаёт все таблицы.
4. Сама заводит компанию и админа (seed, идемпотентный — повтор не ломает данные).
5. Открывается окно входа: `admin` / `admin123`.

Клиент видит просто «программу» (как 1С / кассовые программы) — база спрятана внутри.

**Технические варианты движка (решить при сборке):**
- **A. Встроенный PostgreSQL** (портативные бинарники в комплекте; есть готовая npm-обёртка
  `embedded-postgres` — `initdb` + `pg_ctl` автоматически). ✅ Рекомендуется: 100% совпадает
  с облаком (та же Prisma-схема, те же миграции, ничего не переписываем).
- B. PGlite (Postgres в WASM, в одном процессе, без отдельного бинарника) — новее, легче,
  но надо проверить совместимость с нашими фичами.
- C. SQLite — легче всего, но у нас Postgres-специфичные вещи (частичные unique-индексы,
  типы) → потребует переделок. Менее желателен.

Во **всех** вариантах клиент **не ставит ничего** — база едет внутри установщика.

**Важно для обновлений:** миграции **только аддитивные**, seed **идемпотентный** →
обновление программы **не теряет и не задваивает** данные клиента. (Уже соблюдается в проекте.)

## 4. Локальная сеть — один главный компьютер + кассы

Стандартная и правильная схема для типографии/точки. Один ПК — **главный**
(у руководителя или на складе): держит базу + сервер. Остальные (кассы) подключаются
к нему **по локальной сети**. Кассы **не** держат свою базу — все смотрят в базу главного.

```
                 🖥️ ГЛАВНЫЙ компьютер (руководитель / склад)
                     ├─ внутри: база данных + сервер (API)
                     │
        ┌────────────┼────────────┐   ← локальная сеть (Wi-Fi / кабель)
        ▼            ▼            ▼
     💻 Касса 1   💻 Касса 2   💻 Склад
     (подключены к главному — один склад, одни заказы, одна касса, в реальном времени)
```

Плюс этой схемы: продал на кассе 1 → на складе и у руководителя **сразу** видно.
Остатки, заказы, деньги — **общие, в реальном времени, без синхронизаций внутри офиса**.

**Оговорка (честно):** пока работают кассы, **главный ПК должен быть включён**
(как любой офисный «сервер»). Для одной точки это нормально. Офлайн-устойчивость между
**филиалами** обеспечивает облачная синхронизация (§5).

---

## 5. Облако сверху — филиалы и удалённый доступ руководителя

Главный компьютер филиала синхронизируется с **облаком**. Это даёт:
- руководитель видит всё **удалённо** (из дома, с телефона);
- связь **разных филиалов** между собой.

```
   Филиал 1 (главный ПК + кассы)  ┐
                                  ├──→  ☁️ ОБЛАКО (руководитель видит всё, все филиалы)
   Филиал 2 (главный ПК + кассы)  ┘
```

**Уже готово в проекте:** модуль синхронизации (pull/push, курсоры, HMAC-подпись,
durable nonce против replay) — см. [`docs/03-sync.md`](03-sync.md). Коробка ↔ облако
идёт через него.

## 6. Настройки Electron — выбор роли компьютера

При установке программа спрашивает **роль этого компьютера** (простой выбор, без терминов):

| Выбор | Что делает |
|-------|-----------|
| **«Главный компьютер»** | Держит базу. Остальные подключаются к нему. |
| **«Касса / рабочее место»** | Спрашивает адрес главного ПК (напр. `192.168.1.50`) и подключается. |
| **(галочка) «Синхронизировать с облаком»** | Главный ПК шлёт данные в облако (для руководителя/филиалов). |

Адрес главного ПК программа **сама показывает** на главном компьютере, чтобы на кассах
просто ввести цифры. Никаких конфигов руками.

## 7. Авто-обновление

Клиент **не переустанавливает** программу. Мы выпускаем новую версию → установленные
программы **сами скачивают и применяют** обновление (как браузер / WhatsApp).
Механизм: `electron-updater` + канал раздачи (наш сайт или GitHub Releases).

## 9. Что уже готово, а что предстоит собрать

### ✅ Фундамент уже есть (не с нуля)
- База — **PostgreSQL**, умеет работать по сети (родная функция).
- **Модуль синхронизации** коробка↔облако — написан и работает (`docs/03-sync.md`).
- **Разделение по компаниям/филиалам** (multi-tenant) — есть.
- **Миграции аддитивные, seed идемпотентный** — обновления не теряют данные.
- Веб — **PWA** (мобильный вход уже возможен).
- **Облачный деплой** (Render) — инфраструктура готова и проверена
  ([`render.yaml`](../render.yaml), [`docs/02-deploy-render.md`](02-deploy-render.md)).

### 🔨 Собрать в фазе Electron (новая, но прямая работа)
1. **Скелет Electron**: окно, которое поднимает наш `printpro-api` и показывает `printpro-web`.
2. **Авто-разворачивание встроенной базы** при первом запуске (Postgres внутри, `initdb`,
   миграции, seed) — клиент не ставит ничего.
3. **Экран настроек «роль ПК»**: главный / касса + адрес главного; галочка «облако».
4. **Авто-обновление** (`electron-updater`).
5. **Установщик `.exe`** для скачивания с сайта (упаковка `electron-builder`).
6. **Лицензирование** (Ed25519, привязка к устройству, `license.lic`) — для коробки
   (для облака не нужно). Уже в бэклоге статуса.
