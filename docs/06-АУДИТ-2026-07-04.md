## 1. Executive Summary

**Что уже хорошо (проверено по коду):**
- Деньги хранятся в `Decimal`, не во `Float` — ни одного `Float` в схеме. Округления на уровне БД корректны.
- Ядро кассы вычищено двумя прошлыми итерациями: идемпотентность `quickSale`, атомарная оплата (`updateMany` с guard), защита от двойного открытия смены и дубля штрихкода (частичные unique-индексы), возвраты учитывают ранее возвращённое.
- Минус на складе в производстве устранён (атомарное списание не ниже 0).
- Guard'ы безопасности К1–К3 из прошлого аудита закрыты качественно (services, reports, products/stock — companyId из токена).
- Матрица переходов статуса заказа + история статусов есть.

**Что критично сломано (новое, этой проверкой):**
- **P0 IDOR в модулях `design` и `equipment`** — companyId из query/body, `ensure(id)` без проверки владельца. Сотрудник компании A читает/правит/удаляет чужие макеты и оборудование; через смену статуса чужого макета двигает статус чужого заказа. **Подтверждено прямым чтением кода.**
- **P0/P1 двойной сабмит денежных форм на фронте** — кнопки оплаты/выплаты/погашения долга не блокируются на время запроса; вне POS нет идемпотентности → реальный риск двойного списания денег.
- **P1 расхождения формул в отчётах** — полные возвраты (через refund→CANCELLED) исчезают из выручки, а частичные учитываются; двойной учёт возврата в Z-отчёте; `daily` и `summary` считают выручку из разных источников. Владелец увидит это как «касса/отчёты не сходятся».
- **P1 согласование макета не блокирует печать** — производство запускается по неутверждённому макету тремя обходными путями; нет фиксации «кто утвердил» и версий макета. Для типографии это профильный риск (тираж по несогласованному файлу).

**Ключевые бизнес-риски:** двойное списание денег (фронт), расхождение отчётов с кассой (формулы), кросс-компанийные утечки (design/equipment/roles/search), печать несогласованного тиража, открытая раздача файлов макетов по прямой ссылке.

**Можно ли в прод?** Для одной компании/одной кассы после закрытия P0 (IDOR design/equipment + busy-флаги на денежных формах + формулы возвратов) — да, пилот с одним доверенным оператором. Для мультитенанта (облако, несколько компаний) — **нет, пока не закрыт весь блок IDOR и раздача /uploads.**

### P2 — серьёзные (выборка; полный список — в §приложении по зонам)

- Audit-trail денег не пишет сумму/метод/before-after (audit.interceptor.ts:40-47).
- `settings.setMany` принимает любой ключ/значение без валидации; `bonusAccrualPercent` без верхнего лимита (settings.service.ts:53-65).
- Тумблер `notifyOrderReady` не влияет на реальную рассылку клиенту (orders.service.ts:1170-1191).
- `CashMovement.type` строкой вместо enum; `DocumentCounter` без года (нумерация не сбрачивается 1 января).
- Отчёты режут сутки по timezone сервера, не компании (Render UTC vs Душанбе UTC+5).
- Себестоимость: 3 разных источника цифры за один период (последняя закупка vs снапшот vs текущая цена).
- Недостача материалов при списании в производстве скрывается молча; повторная печать после брака не списывает материалы.
- Нет helmet, CORS открыт всем; JWT_SECRET не валидируется при старте; деактивированный сотрудник работает до истечения токена.
- Дубли заданий на один заказ → двойное списание нормы материалов.
- `daily` игнорирует выбранный период (жёстко 7/30 дней от сегодня).

### P3 — косметика / второстепенное
Округления в JS-number до записи в Decimal; горячие клавиши POS — только надписи; «Быстрый товар» — заглушка; `prompt()` вместо модалок; сортировка колонок в таблицах; маскировка SMTP-пароля в настройках; дубль-timestamp миграции.

## 3. Gap Analysis (чего не хватает для полноценной типографии)

**Производство / макеты:**
- Клиентское согласование пруфа (внешняя ссылка/портал) — сейчас «согласовал» жмёт сам менеджер, доказательной базы нет.
- Версии макета + подпись «кто утвердил, когда».
- Резервирование материалов под заказ (между «принят» и «завершено» материалы ничем не заняты).
- Списание по факту vs по норме; учёт приладки/брака (make-ready waste).
- Выпуск готовой продукции на склад (производство только списывает, готовый тираж на складе не появляется).
- Калькулятор стоимости печати (формат × тираж × цветность × материал).

**Склад:**
- Партии/FIFO/средневзвешенная себестоимость.
- Документ инвентаризации (снимок «учёт на старте», ведомость расхождений со стоимостью).
- Накладная перемещения (тип TRANSFER, статус «в пути», подтверждение приёма).
- Возврат поставщику (частичный); возврат из производства / оприходование брака.
- Справочник причин списания/корректировки.

**Касса / деньги:**
- Частичный возврат по позициям из POS (бэкенд `createReturn` готов — нет UI).
- Возврат долговой продажи из кассы (paid=0 отфильтрован).
- Журнал бонусных операций (`BonusTransaction`) и журнал частичных погашений долга (`ClientDebt` объявлена, но не ведётся).
- Раздельная фиксация недостачи/излишка при закрытии смены.

**Отчёты:**
- Полный отчёт по движению кассы (приход+расход+баланс), отдельный отчёт по возвратам.
- Агрегация week/month (`timeseries` заявлен, но есть только `daily` по дням).
- Выручка по филиалам (мультифилиальность в отчётах не поддержана).

**Настройки:**
- Налоги/НДС отсутствуют полностью (при этом в реквизитах есть ИНН/р.счёт).
- Шаблоны документов (чек/счёт/акт), справочник причин.
- Валидация значений настроек.

## 5. План исправлений

**Быстрые (день-два, малый риск):**
1. IDOR design/equipment/roles/search: заменить `@Query/@Body companyId` на `@CurrentUser().companyId`, в `ensure(id, companyId)` добавить проверку владельца. Убрать `companyId` из входных DTO вообще.
2. Единый паттерн `busy`-флаг + `disabled` + try/finally на всех денежных формах (P0-3, P0-4).
3. `@RequirePermissions` на методах StockController/ProductsController; guard в `createRole`, `setStatus`/`deleteTask`.
4. `deletedAt:null` + `status != CANCELLED` в `salesByItem`, `deletedAt:null` в `profit`/`staffPerformance`/`orders.buildWhead`.
5. Замок `updateMany({status:{not:CANCELLED}})` в начале refund/createReturn; `updateMany` в closeShift.
6. `helmet()` + `enableCors({origin:[...]})`; fail-fast при пустом JWT_SECRET; сверка `isActive` в guard.
7. Проверять `notifyOrderReady` перед рассылкой; убрать `balanceDue` из публичного чека.

**Средние доработки:**
8. Закрыть раздачу `/uploads`: отдавать файлы через авторизованный контроллер с проверкой компании.
9. Единая модель возвратов: сторнирующий Payment соответствующего метода (чинит P1-1, P1-2 разом).
10. Признак `stockDecremented` на Order → корректный restock при возврате.
11. Персональная скидка клиента в `orders.create` (печатные заказы).
12. Согласование как барьер: старт печати требует DesignProof=APPROVED (или явный override с причиной + настройка компании).
13. Валидация ключей/диапазонов в `settings.setMany`.

**Архитектурные (пакетом, с миграциями):**
14. FK `companyId → Company` на ~19 моделей (проверить orphan перед миграцией; Restrict/SetNull, не Cascade).
15. `CashMovementType` enum; `year` в ключе `DocumentCounter`.
16. `BonusTransaction` + реальное ведение `ClientDebt`.
17. `companyId` в `Stock` (или составная проверка владения во всех складских мутациях).
18. TZ компании для границ отчётных периодов.
19. Версионирование макетов (`DesignProofVersion` + approvedBy/approvedAt).

**Удалить:** мёртвые эндпоинты склада, мёртвые поля DTO/настроек, один из дублей POS-скина (после сведения логики).

## 6. Финальный verdict

Платформа функционально богата и за две прошлые итерации денежное ядро и склад заметно укрепили — гонки оплаты, минус-склад, дубли смен/штрихкодов закрыты. Но проверка вскрыла три системных проблемы, которые прошлые аудиты не покрыли:

1. **Мультитенантная защита неполная** — `design`, `equipment`, `roles`, `search` остались с companyId из запроса; для облака это блокер.
2. **Фронт пропускает двойные клики на деньгах** вне POS — прямой риск задвоенных выплат/погашений.
3. **Отчёты и касса расходятся на возвратах** — разные формулы в разных эндпоинтах; владелец увидит «не сходится».

Плюс профильный для типографии дефект: **согласование макета не является барьером** для запуска печати и не оставляет следа «кто утвердил».

Зрелой систему пока считать нельзя — не из-за отсутствия функций, а из-за незакрытого контура целостности (IDOR + формулы возвратов + барьер согласования). После закрытия блоков P0 и P1 система готова к пилоту на одной компании; для мультитенантного облака дополнительно нужен весь IDOR-блок и защита файлов.
