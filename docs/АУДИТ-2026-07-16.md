## Исправлено: критичное и высокое

### 1. Tenant-изоляция синхронизации

Раньше HMAC подтверждал узел, но `pull` всё равно выбирал строки всех компаний,
а `push` мог принять чужой `companyId`. Теперь:

- в облаке обязательны обе пары: `SYNC_NODE_SECRETS=NODE:secret` и
  `SYNC_NODE_COMPANIES=NODE:companyId`;
- узел без tenant-привязки получает 403;
- все 51 модель фильтруется по компании; дочерние модели без `companyId`
  ограничиваются через родителя;
- входящие строки, существующий UUID и tenant-зависимые внешние ключи проверяются
  до upsert;
- глобальный справочник `Permission` для точек read-only;
- legacy shared-secret режим допускается только для БД с одной компанией либо с
  явным `SYNC_COMPANY_ID`;
- добавлены тесты HMAC→company mapping, чужого UUID и дочерних FK.

### 2. Гонка «денежная операция ↔ закрытие смены»

Оплаты, возвраты, расходы, зарплата, авансы и закупки теперь блокируют строку
открытой смены через PostgreSQL `SELECT … FOR UPDATE`. Закрытие использует ту же
блокировку и считает Z-отчёт внутри транзакции. Деньги больше не могут попасть в
смену после уже рассчитанного Z-отчёта.

### 3. Межтенантные idempotency-ключи

Повтор теперь считается тем же запросом только при совпадении компании и целевой
операции/заказа. Коллизия с другой компанией или другим объектом возвращает 409
без раскрытия чужих данных. Исправлены заказы, оплаты, возвраты и складские
операции; Prisma P2002 больше не превращается в непрозрачный 500.

### 4. Привилегии Electron

Рабочее окно больше не получает API настройки/секретов/восстановления базы:

- setup и рабочее окно используют разные preload;
- IPC проверяет конкретного отправителя;
- включены `sandbox` и `contextIsolation`;
- навигация ограничена origin приложения, внешние ссылки разрешены только по
  HTTP(S) и открываются системным браузером.

## Исправлено: данные, инфраструктура и UX

- Sync-курсор не двигается при частично неприменившейся пачке; добавлено
  пятиминутное overlap-окно и атомарная запись state-файла.
- Реестр sync сверен с Prisma-схемой: добавлены ранее пропущенные
  `SupplierPayment`, `PurchaseRequest`, `HeldSale`, `Return` и `WriteOff`;
  тест теперь блокирует последующее расхождение реестра и схемы.
- HMAC подписывает тело + timestamp + nonce; сравнение секретов constant-time.
- Обычные JSON-запросы ограничены 1 МБ, 50 МБ оставлено только для `/api/sync`.
- Параллельные складские приёмки записывают корректные `beforeQty/afterQty` из
  результата атомарного increment.
- Отчёты исключают tombstone-записи заказов, позиций, оплат, движений, смен,
  клиентов и платежей поставщикам; некорректные даты дают 400 вместо 500.
- Исправлены React loading-state, бонусный баланс POS, подписи полей, клавиатурная
  навигация вкладок, семантика поиска/переключателей и skip-link.
- Next обновлён 16.2.9 → 16.2.10. Вложенный уязвимый PostCSS заменён override на

## Обязательные действия перед облачной синхронизацией

В Render → `printpro-api` → Environment для каждого узла добавить обе записи:

```env
SYNC_NODE_SECRETS=K1:<секрет>,K2:<секрет>
SYNC_NODE_COMPANIES=K1:<companyId>,K2:<companyId>
```

Установщик локального узла печатает готовые значения. Если задан
`SYNC_NODE_SECRETS`, но отсутствует соответствующий `SYNC_NODE_COMPANIES`,
синхронизация намеренно блокируется.

## Открытые риски и решения владельца

1. **Публичный кабинет по одному телефону.** Номер телефона сейчас достаточен для
   просмотра ФИО, истории заказов, сумм и долгов, а также повторного заказа.
   Нужен продуктовый выбор: OTP, PIN или magic link. Тихо менять контракт нельзя.
2. **`server-host.js` — незавершённая отдельная работа.** Файл сохранён, но режим
   Windows-службы не подключён к `main.js`/установщику и не включён в пакет.
   Текущая коробка продолжает использовать проверенный встроенный запуск.
3. **Подписанный installer.** `win-unpacked` собран. Обычный шаг `winCodeSign`
   на этой машине блокируется отсутствием права создавать симлинки в кэше.
   Для релизного installer включить Windows Developer Mode/админ-права и
   повторить `npm run dist`; затем провести живой тест установки/обновления.
4. **API lint debt.** Глобальный lint показывает 1418 старых нарушений
   (`any` + форматирование). Это отдельная безопасная серия рефакторинга;
   автоматически переписывать весь API в этом аудите не стали.
5. **Низкое тестовое покрытие.** 23 теста при 31 API-модуле — лучше прежних 16,
   но денежным сценариям нужны интеграционные PostgreSQL-тесты конкуренции.
6. **Production-инфраструктура.** В `render.yaml` всё ещё free DB и эфемерные
   uploads; перед боевыми клиентами нужны платная БД с backup/PITR и постоянное
   файловое/объектное хранилище.
7. **Коробка:** `embedded-postgres` beta и `asar:false` остаются принятыми
   техническими рисками; менять их следует отдельным релизом с живым installer
   и restore-тестом.
