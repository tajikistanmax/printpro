# Синхронизация: локальный узел ↔ облако

> Этап 1 (готов): фундамент + срез клиент→заказ→позиции→оплата.
> Этап 2 (готов): **полный охват — 39 таблиц** (все данные системы синхронизируются).
> Этап 3 (готов): индикатор «онлайн / последняя синхронизация» в панели, heartbeat, мягкие удаления.
> Этап 4 (готов): **установщик локального узла в один клик** (Docker: Postgres + API + панель + синхронизатор).
> Архитектура: облако (Render) — центр; у каждой точки свой локальный сервер; точки
> синхронизируются через облако. Конфликты — «последнее изменение побеждает» по `updatedAt`.

## Как это работает
- В таблицах есть метки `updatedAt` / `deletedAt` / `syncNode`.
- API отдаёт изменения после метки времени: `POST /api/sync/pull { since }`
  и принимает их: `POST /api/sync/push { changes, peer }`. Защита — заголовок
  `x-sync-secret` (значение из переменной `SYNC_SECRET`, одинаковое на узлах).
- При применении сохраняется исходная метка времени — поэтому изменения не «зацикливаются».
- Номера заказов с префиксом узла: `ORD-<NODE_ID>-2025-000001` — не сталкиваются между точками.
- Синхронизатор `scripts/sync-worker.mjs` крутится на локальном узле: раз в N секунд,
  если есть интернет, тянет из облака и отправляет своё.

## Запуск локального узла — в один клик (Этап 4, рекомендуется)
На компьютере точки нужен только **Docker Desktop** (ставится один раз).
1. Скачать/скопировать папку проекта на компьютер точки.
2. Запустить Docker Desktop (значок кита в трее).
3. Двойной клик по **`установить-локальный-узел.cmd`** — установщик спросит:
   - **CLOUD_API** — адрес облака (по умолчанию `https://printpro-api.onrender.com/api`);
   - **SYNC_SECRET** — секрет из облака (Render → printpro-api → Environment);
   - **NODE_ID** — код точки (`K1`, `K2`, `MAGAZIN1`...).
4. Установщик соберёт и поднимет 4 сервиса (Postgres + API + панель + синхронизатор),
   сгенерирует пароль БД и JWT, сохранит всё в `.env.local`, дождётся готовности и откроет панель.

После установки:
- Панель: **http://localhost:3001**, API: **http://localhost:3000/api**.
- БД пустая — первая синхронизация подтянет из облака компанию, пользователей и все данные;
  затем можно входить теми же логинами, что и в облаке. Работает офлайн, при появлении сети — досинхронизирует.
- Логи синхронизации: `docker compose -f docker-compose.local.yml logs -f sync`.
- Остановить: `docker compose -f docker-compose.local.yml down`. Снова поднять:
  `docker compose -f docker-compose.local.yml --env-file .env.local up -d`.
- Данные БД и загрузки хранятся в Docker-томах (`pgdata`, `uploads`) — переживают перезапуск.

> Доступ с других компьютеров точки по локальной сети: панель собрана под `localhost`.
> Для доступа по IP пересоберите `web` с `NEXT_PUBLIC_API_BASE=http://<IP-сервера>:3000/api`.

## Запуск локального узла вручную (без Docker)
1. Поставить PostgreSQL локально, создать базу `printpro`.
2. В `printpro-api/.env`: `DATABASE_URL`, `JWT_SECRET`, и для синхронизации:
   ```
   NODE_ID=K1
   SYNC_SECRET=<тот же секрет, что в облаке (Render → printpro-api → SYNC_SECRET)>
   ```
3. Поднять локальный API: `npm run build && npm run start:prod` (миграции — `prisma migrate deploy`).
4. Запустить синхронизатор:
   ```
   set CLOUD_API=https://printpro-api.onrender.com/api
   set SYNC_SECRET=<секрет>
   set NODE_ID=K1
   npm run sync
   ```
   Курсоры хранятся в `printpro-api/.sync-state.json`.

## Что синхронизируется (Этап 2 — полный охват)
Все 39 таблиц: компании, филиалы, роли/права, пользователи, клиенты, услуги (+тиры/
размеры/опции), товары/склад/движения/приёмки, заказы (+позиции/детали/файлы), касса
(смены/оплаты/движения/долги), оборудование, производство, дизайн-макеты, задачи, настройки, зарплата.
Журнал действий (`AuditLog`) не синхронизируется — он локальный для каждого узла.

Лимит размера запроса поднят до 50 МБ (большие первичные пачки). Курсор — серверная
метка `until`, поэтому расхождение часов узлов не мешает.
