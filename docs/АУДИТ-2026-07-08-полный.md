# Полный аудит + сквозное тестирование PrintPro — 2026-07-08

> Проведён: живой запуск API+Web, HTTP end-to-end тесты, визуальный тест (headless-скриншоты залогиненных страниц), статический аудит кода бригадой из 4 агентов (безопасность, деньги/склад, фронтенд, архитектура/инфра). Учтено уже исправленное из `СТАТУС-ПРОЕКТА.md` — ниже только то, что **осталось**.

## TL;DR
**Ядро продукта — на уровне «почти прод»** (деньги в Decimal, мультитенант-изоляция в бизнес-модуле, идемпотентность операций, RBAC, отзыв токенов, безопасный Electron). Всё запускается и работает: вход, JWT, 22 эндпоинта, все ключевые экраны рендерятся с реальными данными.

**Слабое звено — периметр и процессы:** публичные эндпоинты утекают данные тенанта, sync не скоупится по компании, инфраструктура (CI/тесты/docker) не готова, во фронтенде накопился структурный долг. Критичных data-loss багов в этот заход не найдено, но есть 3 денежных MEDIUM, подрывающих ранее «закрытые» фиксы.

## 2. Безопасность

### 🔴 HIGH
- **Sync без tenant-скоупинга** — `sync/sync.service.ts:95-116` (pull) фильтрует только по `updatedAt`, без `companyId`; `push` (`:119-200`) делает `upsert` с входящим `companyId` без проверки владения. При shared-DB облаке узел с валидным sync-секретом читает/перезаписывает данные **любой** компании. Уже в бэклоге (ждёт решения по топологии) — но остаётся крупнейшим незакрытым риском.

### 🟠 MEDIUM
- **Загрузка файла клиента без фильтра типа** — `clients/clients.controller.ts:86-110`: инлайновый `diskStorage` без `fileFilter`, whitelist MIME/ext и без `toLowerCase()`. Проходят `.html`/`.svg`/`.js` → раздаются статикой как `text/html` → **stored-XSS на origin API**. Единственная из 4 загрузок, пропущенная при харднинге. Плюс путь захардкожен `'./uploads'` вместо `UPLOADS_DIR` (в коробке файлы уйдут мимо каталога статики).
- **`/settings/ui` без авторизации раскрывает платёжные реквизиты** — `settings/public-settings.controller.ts`. ✅ **Подтверждено вживую:** `GET /api/settings/ui?companyId=<id>` без токена вернул `payTransferQr`, `payTransferRequisite`, `companyName`, `phone`, `displayQr`. Это реквизиты, куда клиенты переводят деньги.
- **`/system/company-id` без авторизации выдаёт валидный companyId** — `system/system.module.ts:13-27` (старейшая компания). ✅ **Подтверждено вживую.** Это «ключ», которого не хватает для эксплуатации остальных публичных эндпоинтов.

### 🟡 LOW
- `public/receipt/:id` без scope компании — чек по подобранному id (С16). ✅ вживую: несуществующий id вернул 200.
- Публичный online-заказ не проверяет принадлежность `serviceId` (`public.controller.ts:143-166`).
- `users.create` не проверяет `branchId` на принадлежность компании (`users.service.ts:19-63`).
- `JWT_SECRET` требуется только при `NODE_ENV==='production'` (`main.ts:35-46`) — если NODE_ENV не выставлен в деплое, fail-fast не сработает.

**Позитив:** мультитенант-изоляция в бизнес-модуле (clients/products/stock/categories/branches/equipment/design/cash/notifications/complaints/settings/audit/reports/users/roles) — единообразна и корректна; `companyId` строго из JWT; FK-проверки владения повсеместны; raw-SQL параметризован; `JWT_SECRET` без слабого дефолта.

## 3. Деньги и склад

### 🟠 MEDIUM
- **Гонка кредит-лимита не закрыта** — `orders.service.ts:200-218`. Проверка перенесена внутрь транзакции, но без `SELECT … FOR UPDATE` под READ COMMITTED две параллельные транзакции читают `currentDebt=0` и обе проходят лимит → долг превышает лимит. **С14 в статус-доке помечен «закрыт» ошибочно.**
- **Потеря бонусов при возврате** — `orders.service.ts:1046-1057` (refund) и `:1365-1376` (createReturn): сторнирование бонусов написано как абсолютная перезапись (`update: { bonusPoints: newBonus }`), а не atomic `{ decrement }` как в quickSale. Параллельный возврат+начисление теряет баллы (деньги-эквивалент).
- **`addPayment` без идемпотентности** — `order-actions.dto.ts:15-20` / `orders.service.ts:397-491`. Идемпотентность есть у create/quickSale/return, но не у обычной оплаты. Ретрай после потерянного ответа создаёт вторую оплату → двойное начисление денег и бонусов.

### 🟡 LOW
- Явный `shiftId` в оплате проверяется без `userId` (`orders.service.ts:443-453`) → платёж в чужую открытую смену → искажение атрибуции Z-отчёта.
- Импорт CSV не отбраковывает отрицательные цены (`products.service.ts:212-219`) — минует `@Min(0)` DTO.

**Позитив:** С7 (операции без смены) **фактически закрыт в коде** (статус-док стоит обновить); транзакционность денежных/складских операций и `recordTx`-аудит — на месте; склад не уходит в минус (условный `updateMany`); Decimal-арифметика с epsilon-сравнениями; формулы отчётов согласованы.

## 4. Фронтенд (printpro-web)

### 🔴 HIGH
- **Нет общего data-хука** — паттерн `api.get().then(setX).catch(()=>{})` скопирован в ~30 файлах; из-за этого массово нет loading/error-состояний, а `companyId` вручную вшит в **149 fetch-вызовов** в 32 файлах.
- **Проглоченные `catch {}` на мутациях** — `services/page.tsx:233` (удаление услуги), `design/page.tsx:162` (чек-лист), `warehouse/page.tsx:313`, `pos/page.tsx:685,692`: операция молча проваливается без сообщения.
- **366 `any`** в 34 файлах; ответы API нигде не типизированы; центральный `PosCtx` (`_pos.tsx:34-49`) весь на `any[]`.
- **JWT в localStorage** (`auth.tsx:74,85`, читается в `api.ts:5` и вручную в 3 местах для multipart) — доступен любому XSS. Известно, в бэклоге (httpOnly-cookie).

### 🟠 MEDIUM
- Монолиты: `_pos.tsx` (1723 стр.), `reports/sections.tsx` (1685), `settings/page.tsx` (1498), `purchasing/page.tsx` (1186), `warehouse/page.tsx` (1004).
- `alert()` вместо UI — 8 мест (`staff`, `warehouse`, `purchasing`).
- 403 не обрабатывается централизованно (только 401 → `handleUnauthorized`, `api.ts:9`).
- Raw-fetch загрузки (`clients/page.tsx:221`, `production/page.tsx:135`) минуют `handleUnauthorized` — при 401 не разлогинивают.
- POS-оплата без `disabled` на кнопках (`pos/page.tsx:526`) — двойной клик защищён только backend-идемпотентностью, визуально не предотвращён.
- 100% `'use client'` (53/53 файла) — SSR/RSC/metadata не используются; публичные страницы теряют SEO.
- Экспорт заказов тянет все записи `pageSize=total` (`orders/page.tsx:300`).

**Позитив:** централизованный `api.ts` с корректным 401-флоу; пагинация на ключевых списках; security-заголовки; экраны ошибок; cleanup всех интервалов; идемпотентность POS; нет `@ts-ignore` и утечек токена в URL.

## 5. Архитектура и инфраструктура

### 🔴 HIGH
- **Нет CI** — в `.github/workflows` только `db-backup.yml`. Ничего не гоняет tsc/build/test/lint на push/PR. Регрессии денег ловятся только вручную.
- **Почти нет тестов** — 1 содержательный spec (`order-math.spec.ts`) на 135 файлов api; на web — 0. Транзакционные сервисы (оплаты/возвраты/производство/payroll/sync) не покрыты.
- **Docker-путь локального узла нерабочий для многомашинного сценария:**
  - `printpro-web/Dockerfile:8-9` зашивает `NEXT_PUBLIC_API_BASE=http://localhost:3000/api` → браузер кассы на другой машине пойдёт в свой localhost (где API нет) вместо главного ПК. Ломает LAN-автоопределение.
  - `printpro-api/Dockerfile:25` — `CMD` без seed (вопреки комментарию на `:24`) → свежий узел получает пустую схему без `admin`/ролей, войти нельзя.
- **Нет workspace-корня** (корневого `package.json` нет) → версии зависимостей и константы (`FALLBACK_COMPANY_ID` живёт в 3 местах: `config.ts:62`, `seed.ts:11`, `render.yaml:80`) дублируются и разъезжаются. Нет общих типов api↔web.
- **Отсутствуют индексы под горячие FK** — `OrderItem` (нет ни одного `@@index`), `StockReceiptItem`, `QuoteItem`, `ClientDebt`, `ServicePriceTier/Size/Option`, `AuditLog` (растёт быстрее всех).

### 🟠 MEDIUM
- `CashMovement.type` — `String // IN/OUT` вместо enum (рядом есть `StockMovementType`). Опечатка тихо исказит Z-отчёт.
- `companyId String` без FK на `Company` в большинстве финансовых моделей (Payment, CashMovement, CashShift, Return, WriteOff…) — нет ссылочной целостности тенанта на уровне БД.
- Две миграции с идентичным таймстампом `20260702120000_*` (document_counter + purchase_request).
- `render.yaml` — все прод-сервисы на `plan: free` (Postgres удаляется ~через 30 дней, uploads на эфемерной ФС).
- `asar: false` (`electron/package.json:31`) — весь исходник api/web на диске клиента открыт; устаревшие комментарии про `app.asar.unpacked`.
- `embedded-postgres: 17.10.0-beta.17` — **бета** встроенной БД у клиентов.
- Sync docker всегда в legacy-режиме (`sync-worker.mjs:46` шлёт только `x-sync-secret`, без HMAC/nonce; сравнение секрета нетаймсейф) → заявленная replay-защита P0-5 на поставляемом воркере не действует.
- `.kiro/` и `.claude/` замусорены нерелевантными дизайн-скиллами (banner/brand/slides/ui-ux-pro-max) + закоммичены `.pyc`. Не в `.gitignore`.
- `db-backup.yml` — джоб запускается по крону и падает на проверке секретов → регулярные красные прогоны (комментарий обещает «не запустится»).

### 🟡 LOW
- Разнобой версий Postgres: docker `18-alpine`, embedded `17.x`, backup-workflow `client-16`, Render свой — риск несовместимости дампов.
- `docker api` без `NODE_ENV=production` → Swagger открыт, CORS в dev-ветке.

**Позитив:** Decimal везде (ни одного Float для денег); Electron — `contextIsolation:true`, `nodeIntegration:false`, deny внешних окон, порт-guard'ы, бэкапы; единый идемпотентный seed (не задвоен); прод-хардненинг API (helmet/request-id/fail-fast/graceful shutdown/throttler).
