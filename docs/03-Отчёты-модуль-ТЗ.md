# ТЗ — Модуль отчётов PrintPro

> Платформа для типографии/полиграфии (Таджикистан). Валюта — **TJS (сомони)**, язык интерфейса — **русский**. Все суммы округляются до **2 знаков**, количества — до 3 знаков, проценты — до 1 знака. Даты — **ISO-строки**. Часовой пояс — **серверное время (UTC)**; TJ = UTC+5 — известное упрощение (см. §8).

Документ описывает модуль отчётов: назначение, роли и доступ, глоссарий метрик с точными формулами, каталог из 20 отчётов, фильтры и период, экспорт, бизнес-процессы, краевые случаи и справочник REST API.

## 1. Назначение и цели модуля

Модуль отчётов даёт владельцу и руководству типографии единую, достоверную и сопоставимую картину бизнеса: деньги, выручку, прибыль, долги, склад, закупки, производство и работу персонала — по компании и в разрезе филиалов, за произвольный период.

**Цели:**

- **Контроль денег.** Ежедневное закрытие кассы (Z-отчёт), сверка полученного по способам оплаты, отслеживание кассовых разрывов.
- **Управление прибылью.** Видеть валовую прибыль и маржу по заказам, услугам, товарам, категориям и клиентам; находить убыточные позиции и «утечку маржи» через скидки.
- **Работа с долгами.** Дебиторка (кто нам должен) и кредиторка (кому должны мы) с aging-анализом просрочки.
- **Склад и закупки.** Оценка остатков, дефицит, списания, расход материалов, история закупок и цен поставщиков.
- **Производство и персонал.** Загрузка станков, сроки (lead time), процент брака, эффективность сотрудников.
- **Сопоставимость.** Сравнение с предыдущим периодом (месяц к месяцу), динамика по дням/неделям/месяцам.

**Принципы достоверности** (детально в §3 и §8):

- Товарно-выручочные метрики (выставлено, чистая выручка, прибыль, продажи) считаются **только по заказам со `status != CANCELLED`**.
- Денежные метрики (получено, приток) считаются **по фактическим платежам** `Payment` с `method != DEBT`. Запись «в долг» (DEBT) — это **не деньги**.
- Каждая сумма учитывается **один раз** — возвраты не вычитаются дважды, join'ы не размножают суммы.

## 2. Роли и доступ

Все эндпоинты модуля защищены guard'ами `JwtAuthGuard` + `PermissionsGuard` и требуют право **`reports.view`**. Мультитенантность: любые данные фильтруются по `companyId` (обязательный query-параметр).

| Роль | Доступ | Что смотрит на практике |
|---|---|---|
| **Директор / владелец** | Полный (`reports.view`) | Все отчёты. Пульс бизнеса, месячный финитог с `compare`, прибыль, производство, дебиторка. |
| **Бухгалтер / финансист** | Полный (`reports.view`) | Касса (summary, cashflow), расходы, дебиторка/кредиторка, закупки, реестр заказов для сверки и налоговой. |
| **Менеджер по работе с клиентами** | `reports.view` | Дебиторка (receivables), продажи по клиентам, реестр заказов; звонки должникам. |
| **Начальник производства** | `reports.view` | Производство, загрузка оборудования, staff (брак/сроки). |
| **Кладовщик** | `reports.view` | Склад (inventory), движения (stock-movements), закупки. |
| **Кассир / администратор смены** | `reports.view` (ограниченно) | Свою смену: summary и cashflow за день для Z-отчёта. |

> Разграничение по ролям на уровне отдельных отчётов сейчас не реализовано (единое право `reports.view`). Тонкая настройка видимости конкретных отчётов по ролям — в §10 «Дальнейшее развитие». Ограничение по филиалу для сотрудника точки — через фильтр `branchId`.

## 3. Глоссарий метрик и формулы

Единые формулы соблюдаются во всех отчётах. **Метод начисления** (по дате `Order.createdAt`) — для выручки/прибыли/долга. **Кассовый метод** (по дате `Payment.createdAt`) — для полученных денег. Эти два потока **не обязаны совпадать**: сегодняшний платёж может гасить заказ прошлого месяца.

### 3.1 Выручка и деньги

| Метрика | Определение | Формула |
|---|---|---|
| **Выставлено** (`billed`) | Валовая сумма по заказам периода | `Σ Order.total`, заказы с `createdAt ∈ период`, `status != CANCELLED` |
| **Возвраты** (`returns`) | Контр-выручка (возвращённое клиенту) | `Σ Order.returnedTotal` по тем же заказам |
| **Чистая выручка** (`net`) | Выставлено за вычетом возвратов | `billed − returns` |
| **Получено деньгами** (`collected`) | Реально поступившие деньги | `Σ Payment.amount` где `method != DEBT`, `createdAt ∈ период` |
| **По способам** (`byMethod`) | Разбивка платежей | группировка `Payment.amount` по `method` (cash/card/qr/transfer/**debt отдельно**) |
| **Средний чек** (`avgCheck`) | Средняя сумма заказа | `ordersCount > 0 ? billed / ordersCount : 0` |
| **Инкассация** (`cashCollectionRate`) | Индикатор сбора денег | `net > 0 ? collected / net * 100 : 0` (индикатор, **не тождество**) |
| **Прирост долга** (`debtGrowth`) | Копится ли долг за период | `billed − returns − collected` |

> **DEBT никогда не суммируется в «получено».** Иначе выручка задвоится: сначала запись долга, потом его погашение реальными деньгами.

### 3.2 Прибыль

| Метрика | Определение | Формула |
|---|---|---|
| **Себестоимость** (`cost`) | Затраты на проданное | `Σ (OrderItem.lineCost) − Order.returnedCost` по заказам периода (`status != CANCELLED`) |
| **Валовая прибыль** (`grossProfit`) | Выручка минус себестоимость | `(Σ lineTotal − returnedTotal) − (Σ lineCost − returnedCost)` |
| **Маржа** (`margin`) | Рентабельность по выручке | `revenue > 0 ? profit / revenue * 100 : 0` |
| **Доля без себестоимости** (`zeroCostShare`) | Насколько маржа недостоверна | `net > 0 ? (revenue позиций где lineCost=0) / net * 100 : 0` |
| **Флаг `zeroCost`** | Позиция без себестоимости | `lineCost = 0 && lineTotal > 0` |

> **zeroCost:** услуга/товар без себестоимости даёт маржу ~100% и завышает прибыль. Фиктивную себестоимость **не подставляем** — только помечаем флагом, чтобы недостоверность была видна.

### 3.3 Долги и aging

| Метрика | Определение | Формула |
|---|---|---|
| **Дебиторка** (`receivables`) | Нам должны клиенты | `Σ Order.balanceDue` где `balanceDue > 0`, `status != CANCELLED` |
| **Кредиторка** (`payables`) | Мы должны поставщикам | `Σ Supplier.debt` где `debt > 0` |
| **Aging (старение)** | Разбивка долга по просрочке | по `debtDueDate || createdAt` (дебиторка) / `StockReceipt.dueDate` (кредиторка): `current` (не просрочен), `d1_30`, `d31_60`, `d60plus` |
| **Просрочено всего** (`overdueTotal`) | Сумма просроченного | `Σ debt` где срок оплаты `< now` |
| **Средняя просрочка** (`avgDaysOverdue`) | Средние дни просрочки | среднее `daysOverdue` по просроченным (0 если нет) |
| **Оценка безнадёжного** (`badDebtEstimate`) | Резерв на списание | `d31_60 * 0.3 + d60plus * 0.7` |

### 3.4 ABC-анализ

Сортировка позиций по `revenue` по убыванию; кумулятивная доля нарастающим итогом. Класс: **A** — до 80%, **B** — 80–95%, **C** — >95%. Вторым проходом — `profitClass` по прибыли с теми же порогами.

### 3.5 Склад и оборачиваемость

| Метрика | Определение | Формула |
|---|---|---|
| **Оценка склада** (`inventoryValue`) | Стоимость остатков по себестоимости | `Σ Stock.quantity * Product.purchasePrice` (purchasePrice null → 0) |
| **Потенциальная прибыль** (`potentialProfit`) | Заработок при распродаже остатков | `Σ qty * (salePrice − purchasePrice)` |
| **Средний дневной расход** (`avgDailyUse`) | Расход материала в день | `Σ StockMovement OUT qty за период / дней в периоде` |
| **Запас в днях / оборачиваемость** (`daysOfCover`) | На сколько дней хватит | `avgDailyUse > 0 ? qty / avgDailyUse : null` (**null**, не Infinity) |
| **Процент списания** (`writeOffRate`) | Доля брака/списаний | `inQty > 0 ? writeOffQty / inQty * 100 : null` |
| **Дефицит** (`lowStock`) | Заканчивается | `qty <= minStock && minStock > 0` |
| **Ноль остатка** (`outOfStock`) | Активный расходник в ноль | `qty <= 0 && isActive` |
| **Отрицательный остаток** (`negativeStock`) | Ошибка учёта | `qty < 0` (в totalValue учитываем как есть, выводим предупреждением) |

### 3.6 Производство

| Метрика | Определение | Формула |
|---|---|---|
| **Lead time** (`avgLeadTimeHours`) | Средний срок выполнения | среднее `(completedAt − (startedAt || createdAt))` в часах по COMPLETED; **отрицательные отбрасывать** |
| **Процент брака** (`reworkRate`) | Доля переделок | `(completed + rework) > 0 ? rework / (completed + rework) * 100 : 0` |
| **В срок** (`onTimeRate`) | Уложились в дедлайн | `completed > 0 ? (completed где completedAt <= order.deadline) / completed * 100 : 0` |
| **Просрочено в работе** (`overdueInWork`) | Активные задания с `deadline < now` | count активных ProductionJob |
| **Без исполнителя/станка** (`unassignedJobs`) | Не распределено | активные где `equipmentId = null` OR `assignedUserId = null` |

### 3.7 Закупки

| Метрика | Формула |
|---|---|
| **Средняя приёмка** (`avgReceipt`) | `receiptsCount > 0 ? total / receiptsCount : 0` |
| **По статусам** (`byStatus`) | сумма `StockReceipt.total` по `paymentStatus` {paid, partial, debt} |
| **Средняя закупочная цена товара** (`avgCost`) | `qty > 0 ? amount / qty : 0` |

### 3.8 Универсальное правило деления на ноль

Всегда **0** при знаменателе ≤ 0: avgCheck, margin, sharePct, reworkRate, onTimeRate, avgReceipt. **Исключение:** `daysOfCover`, `turnover` при нулевом расходе → **null** (не Infinity/NaN).

## 4. Каталог отчётов

Общие query-параметры для отчётов «за период»: `companyId` (обязательный), `from?`, `to?` (ISO), `branchId?`, `groupBy?` (`day`|`week`|`month`). Экспорт для всех — CSV и Печать/PDF (§6). Отчёты **3 (daily)** и **12 (debts)** сохраняются без изменений для обратной совместимости.

### 4.1 summary — Финансовая сводка

| | |
|---|---|
| **Назначение** | Ключевые деньги и выручка за период — пульс бизнеса. |
| **Источник** | `Order` (агрегат), `Payment` (groupBy method), `CashShift` (открытые смены). |
| **Фильтры** | from, to, branchId, `compare` (=1 → предыдущий период той же длины). |
| **Столбцы** | ordersCount, billed, returns, net, collected, debt, avgCheck, expensesTotal, grossProfit, margin, newClients, byMethod{cash,card,qr,transfer,debt}, **cashCollectionRate, debtGrowth, zeroCostShare, openShiftsCount**, compare{...deltas}. |
| **Формулы** | §3.1, §3.2. compare — те же метрики за период `[from−длина; from−1день]` + процентные дельты. |

### 4.2 timeseries — Динамика по бакетам

| | |
|---|---|
| **Назначение** | График оборота/денег/прибыли по дням/неделям/месяцам. |
| **Источник** | `Order`, `Payment` по `createdAt`. |
| **Фильтры** | from, to, branchId, groupBy. |
| **Столбцы** | buckets[]: date (начало бакета), label, collected, billed, ordersCount, **profit, debt, avgCheck, cumCollected**. |
| **Формулы** | profit = `(lineTotal−returnedTotal)−(lineCost−returnedCost)` по заказам бакета; debt = `billed−collected`; cumCollected — нарастающий итог. **Пустые бакеты присутствуют** (нули). |

### 4.3 daily — Выручка по дням *(без изменений)*

Массив `[{date, amount}]` за N дней (`days`, по умолчанию 14). `amount` = `collected` (payments `method != DEBT`) по дням. Обратная совместимость — не менять.

### 4.4 sales-by-item — Продажи по позициям

| | |
|---|---|
| **Назначение** | Топ услуг/товаров по выручке и марже. |
| **Источник** | `OrderItem` заказов периода (`status != CANCELLED`). |
| **Фильтры** | from, to, branchId, `type` (SERVICE\|PRODUCT\|all), `categoryId` (по категории или её родителю). |
| **Столбцы** | key, name, type, category, qty, revenue, cost, profit, margin, sharePct, **orders, avgPrice, avgCost, zeroCost**. |
| **Формулы** | Ключ агрегации — по `serviceId`/`productId` (строки без ссылки → `desc:`+description), **не** по имени. avgPrice = `revenue/qty`, avgCost = `cost/qty`. Сорт по revenue убыв. |

### 4.5 sales-by-category — Продажи по категориям

| | |
|---|---|
| **Назначение** | Структура продаж по верхнему уровню категорий с раскрытием подкатегорий. |
| **Источник** | `OrderItem` + `Service/Product` категории (двухуровневые, parentId). |
| **Фильтры** | from, to, branchId, type. |
| **Столбцы** | total, items[]: categoryId, category, type, qty, revenue, cost, profit, margin, sharePct, children[]{...}. |
| **Формулы** | Группировка по верхнему уровню; без категории или с висячим parentId → **«Без категории»**. |

### 4.6 sales-by-client — Продажи по клиентам

| | |
|---|---|
| **Назначение** | Ценность клиентов, кто приносит выручку, кто пропал. |
| **Источник** | `Order` + `Client`. |
| **Фильтры** | from, to, branchId, `limit`. |
| **Столбцы** | clientId, client, phone, orders, revenue, paid, debt, avgCheck, sharePct, **lastOrderDate, daysSinceLastOrder, overdueDebt**. |
| **Формулы** | revenue = `Σ (order.total − returnedTotal)` (`status != CANCELLED`). Заказы без клиента → строка **«Розница/без клиента»**. Сорт по revenue убыв. |

### 4.7 abc — ABC-анализ

| | |
|---|---|
| **Назначение** | Классификация позиций A/B/C по вкладу в выручку и прибыль. |
| **Источник** | агрегированные продажи (как sales-by-item). |
| **Фильтры** | from, to, branchId, type. |
| **Столбцы** | items[]: key, name, type, revenue, sharePct, cumPct, class, **profit, profitClass**; summary{A,B,C}{count,revenue,sharePct}. |
| **Формулы** | §3.4. |

### 4.8 profit — Прибыль по заказам

| | |
|---|---|
| **Назначение** | Прибыль и маржа по каждому заказу, поиск убыточных. |
| **Источник** | `Order` + `OrderItem` (`status != CANCELLED`). |
| **Фильтры** | from, to, **branchId** (добавлен). |
| **Столбцы** | ordersCount, revenue, cost, profit, margin, **totalReturns, zeroCostShare**, items[]{orderId, orderNumber, date, client, revenue, cost, profit, margin, **loss**}. |
| **Формулы** | §3.2. loss = `profit < 0`. Отрицательные revenue/cost/profit не обрезать в 0 (только margin при revenue≤0 → 0). |

### 4.9 expenses — Расходы кассы

| | |
|---|---|
| **Назначение** | Структура операционных расходов по категориям. |
| **Источник** | `CashMovement` type=OUT, **исключая** category=«Возвраты». |
| **Фильтры** | from, to, branchId (через shift). |
| **Столбцы** | total, byCategory[]{category, amount, **count, sharePct**}, items[]{date, category, reason, amount}. |
| **Формулы** | Возвраты — контр-выручка (уже в net), в расходы **не включаем** (иначе двойной счёт). Сравнение категории по trim/регистру. |

### 4.10 cashflow — Движение денег

| | |
|---|---|
| **Назначение** | Приток/отток денег, сверка кассы, кассовые разрывы. |
| **Источник** | `Payment` (приток), `CashMovement` OUT + `SupplierPayment` (отток), `CashShift`. |
| **Фильтры** | from, to, branchId (через shift), groupBy. |
| **Столбцы** | inflow, outflow, net, byMethod{cash,card,qr,transfer}, outByCategory[], **supplierPayments, refundsCash, openingBalance, closingBalance, expectedClosing, discrepancy, openShiftsCount**, buckets[]{date, label, inflow, outflow, net, **cumNet**}. |
| **Формулы** | inflow = `Σ Payment где method != DEBT`; outflow = `Σ CashMovement OUT (excl Возвраты) + Σ SupplierPayment`. refundsCash — отдельным полем, **не в outflow**. Незакрытую смену не считать closing=0 — помечать `openShiftsCount`. |

### 4.11 receivables — Дебиторка

| | |
|---|---|
| **Назначение** | Кто нам должен, aging, работа с должниками. |
| **Источник** | `Order` где `balanceDue > 0`. |
| **Фильтры** | branchId. |
| **Столбцы** | total, count, aging{current, d1_30, d31_60, d60plus}, items[]{orderId, orderNumber, client, phone, total, paid, debt, dueDate, overdue, daysOverdue}, **overdueTotal, avgDaysOverdue, topDebtors, badDebtEstimate**. |
| **Формулы** | §3.3. aging по `debtDueDate || createdAt`. topDebtors — первые 5 по debt. |

### 4.12 debts — Долги клиентов *(без изменений)*

`{total, count, items[]{orderId, orderNumber, client, phone, total, paid, debt}}`. Обратная совместимость — не менять. Для aging использовать **receivables** (4.11).

### 4.13 payables — Кредиторка

| | |
|---|---|
| **Назначение** | Кому и сколько должны мы, что просрочено. |
| **Источник** | `Supplier.debt > 0` + ближайший `StockReceipt` с `paymentStatus != PAID`. |
| **Фильтры** | companyId. |
| **Столбцы** | total, count, items[]{supplierId, supplier, phone, debt, dueDate, overdue, **paidAmount**}, **aging{current,d1_30,d31_60,d60plus}, overdueTotal**. |
| **Формулы** | aging по `StockReceipt.dueDate`, сумма `(total − paidAmount)`. overdueTotal = `Σ где dueDate < now`. |

### 4.14 purchasing — Закупки

| | |
|---|---|
| **Назначение** | Сколько закупили, у кого, по каким ценам. |
| **Источник** | `StockReceipt` по полю `date` + позиции. |
| **Фильтры** | from, to, branchId, `supplierId`. |
| **Столбцы** | total, paid, debt, receiptsCount, bySupplier[]{supplierId, supplier, receipts, total, paid, debt}, topProducts[]{productId, name, qty, amount, **avgCost, lastReceiptDate, priceTrendPct**}, **avgReceipt, byStatus{paid,partial,debt}**. |
| **Формулы** | §3.7. |

### 4.15 inventory — Остатки склада

| | |
|---|---|
| **Назначение** | Стоимость остатков, дефицит, проблемные позиции. |
| **Источник** | `Stock` где `product.deletedAt = null && isActive = true`. |
| **Фильтры** | branchId. |
| **Столбцы** | totalValue, totalSku, totalQty, lowStock[]{productId, name, qty, minStock, unit}, items[]{productId, name, category, unit, qty, purchasePrice, value, **daysOfCover**}, **potentialProfit, outOfStock[], negativeStock[]**. |
| **Формулы** | §3.5. value = `qty * purchasePrice`. |

### 4.16 stock-movements — Движения склада

| | |
|---|---|
| **Назначение** | Реальный расход материалов, списания, приходы. |

| **Формулы** | §3.5. Исторические движения по удалённому/неактивному товару **сохранять** (иначе прошлые периоды занизятся). |

|---|---|

## 5. Фильтры и период

### 5.1 Период

- **Диапазон включительный:** `[from 00:00:00.000, to 23:59:59.999]` по серверному времени.
- **По умолчанию** (нет `from`): текущий месяц с 1-го числа по `now`.
- **Пресеты** (фронт): Сегодня, Вчера, 7 дней, 14 дней, Текущий месяц, Прошлый месяц, Квартал, Год, Произвольный диапазон.

### 5.2 Филиал (`branchId`)

- Пусто/отсутствует = **все филиалы**.
- Orders/items/receipts/stock/writeoff/production — по своему `branchId`.
- Payments — через `order.branchId` (внезаказовый платёж при активном фильтре исключается — см. §8).
- CashMovement / выручка смены — через `shift.branchId`.
- **Правило:** сумма по всем филиалам ≥ суммы по одному филиалу — это не баг.

### 5.3 Гранулярность (`groupBy`)

`day` | `week` | `month` — для timeseries и cashflow. Все бакеты присутствуют, пустые = 0.

### 5.4 Сравнение периодов (`compare=1`)

Для summary: предыдущий период той же длины, вплотную перед `from`. Возвращаются те же метрики + процентные дельты (`collectedPct`, `billedPct`, `netPct`, `grossProfitPct`, `ordersCountPct`, `avgCheckPct`).

## 7. Бизнес-процессы

### 7.1 Ежедневное закрытие кассы (Z-отчёт)

**Кто:** кассир/администратор смены; контролирует директор. **Когда:** каждый вечер при закрытии смены.
1. `summary` за сегодня — сверить collected и byMethod (cash/card/qr/transfer) с фактом.
2. `cashflow` за сегодня — наличный inflow против остатка в ящике (opening + cash-in − cash-out = closing).
3. Проверить `refundsCash` — все ли возвраты денег оформлены документами.
4. Свести `closingBalance` vs `expectedClosing`; при недостаче/излишке — разбор по `staff.collected`.
5. Зафиксировать closingBalance смены.

### 7.2 Утренняя планёрка по производству

**Кто:** начпро + операторы + директор. **Когда:** ежедневно перед сменой.
1. `production` — очередь (inQueue), в работе (inWork), overdueInWork, dueSoon.
2. Срочные (URGENT/EXPRESS) — первыми (`urgentInWork`).
3. Найти узкое место по стадиям, перекинуть людей.
4. Распределить `unassignedJobs` (без станка/исполнителя).
5. `equipment-load` — учесть станки в ремонте (REPAIR/OFF).

### 7.3 Работа с должниками (дебиторка)

**Кто:** менеджер/бухгалтер/директор. **Когда:** 1–2 раза в неделю, чаще в конце месяца.
1. `receivables` — aging с d60plus и d31_60, topDebtors, overdue.
2. `sales-by-client` — оценить ценность клиента (не пережать корп-клиента).
3. Обзвонить крупнейших и самых просроченных, договориться о дате.
4. После оплаты долг уходит из receivables (сверка collected следующего дня).
5. В конце месяца — `badDebtEstimate` для списания безнадёжных.

### 7.4 Инвентаризация и заявка на закупку

**Кто:** кладовщик + бухгалтер + начпро, утверждает директор. **Когда:** еженедельно (дефицит), ежемесячно/квартально (полная).
1. `inventory` — totalValue, lowStock, outOfStock, negativeStock; сверка qty с фактом.
2. По `daysOfCover` — чего хватит на 2–3 дня → в срочную заявку.
3. `stock-movements` — реальный расход, writeOffs и writeOffRate по операторам.
4. `purchasing` — последняя дата прихода, интервал, priceTrend по поставщику.
5. Расхождение факт vs учёт → ADJUST; заявка с учётом lead time поставки.

### 7.5 Недельный разбор (оперативка)

**Кто:** директор + старший менеджер + мастер. **Когда:** понедельник утром.
1. `timeseries` groupBy=day за неделю — динамика collected/billed/прибыли, cumulative.
2. `sales-by-item` + `abc` — топ и убыточные позиции, где сливается маржа (проверить zeroCost).
3. `staff` — у кого объём есть, а прибыль/скидки/reworkRate не в порядке.
4. `production` — onTimeRate, reworkRate, просроченные, throughput.
5. Раздать задачи: догнать по срокам, поднять цену на убыточную услугу, обзвонить долги.

### 7.6 Расчёты с поставщиками (кредиторка)

**Кто:** бухгалтер. **Когда:** еженедельно/по графику.
1. `payables` — кто ждёт оплаты, aging, overdueTotal.
2. `purchasing?supplierId` — история поставок и оплат, byStatus.
3. Спланировать платежи по остатку кассы (`cashflow`).
4. Не допустить просрочки, влияющей на поставки; после оплаты SupplierPayment → cashflow.outflow.

### 7.7 Месячный финансовый итог

**Кто:** директор/финдиректор/бухгалтер. **Когда:** 1–3 число следующего месяца.
1. `summary?compare=1` — net, grossProfit, margin, avgCheck, cashCollectionRate месяц-к-месяцу.
2. `expenses` — структура расходов (count, sharePct), отклонения.
3. `cashflow` — чистый поток, кассовые разрывы; `profit` — топ прибыльных/убыточных.
4. `purchasing` + `payables` — сколько закупили, кому должны; `inventory` — стоимость остатков на конец месяца.
5. `orders` — выгрузка реестра (с returned) для учёта/сверки; выводы: план, цены, лимиты долга, скидки.

### 7.8 Пульс бизнеса (дашборд владельца)

**Кто:** владелец, часто с телефона. **Когда:** ежедневно.
1. `summary` за сегодня + `daily` за 14 дней.
2. cashCollectionRate и debtGrowth — получаем деньги или растёт долг.
3. `summary?compare=1` — лучше или хуже прошлого периода.
4. Тревожный сигнал → провалиться в receivables/production/profit.

## 8. Краевые случаи и правила без двойного счёта

1. **CANCELLED повсюду.** billed/net/grossProfit/avgCheck/sales-by-* — только `status != CANCELLED`. Это устраняет расхождение старого `summary()` (агрегировал все) и `profit()` (уже фильтровал CANCELLED).
2. **Кассовый vs метод начисления.** collected/inflow — по `Payment.createdAt`; billed/net/debt — по `Order.createdAt`. Не приравнивать в UI; для сверки — `cashCollectionRate` как индикатор.
3. **DEBT — не деньги.** collected/inflow/staff.collected/cashflow.inflow — строго `method != DEBT`. DEBT — отдельной строкой `byMethod.debt`, никогда не в «получено».
4. **Возврат — один раз.** `returns` (returnedTotal) уже вычтен из net. CashMovement category=«Возвраты» исключать из expenses и cashflow.outByCategory. В cashflow физический отток по возвратам — отдельным полем `refundsCash`, не в outflow.
5. **Деление на ноль.** §3.8. daysOfCover/turnover при нулевом расходе → null.
6. **zeroCost.** Не подставлять фиктивную себестоимость — флаг + `zeroCostShare`.
7. **Удалённые/неактивные.** Текущие остатки — только `deletedAt=null && isActive=true`. Но исторические продажи/движения по удалённому товару — **сохранять** (иначе прошлые периоды занизятся). Не фильтровать OrderItem по текущему isActive.
8. **Ключ агрегации sales-by-item** — по `serviceId`/`productId` (без ссылки → `desc:`+description), не по имени: одинаковые имена разных товаров не склеивать.
9. **Задания без станка.** equipment-load берёт `equipmentId != null`, поэтому теряет их. В production — `unassignedJobs`, чтобы jobsTotal сходился.
10. **Заказ без клиента.** В profit/debts/receivables/orders — «без клиента». В sales-by-client — отдельная строка «Розница/без клиента».
11. **Без категории.** Товар/услуга без категории или с висячим parentId → «Без категории» верхнего уровня. Строку не терять.
12. **Только Payment для денег.** Денежные агрегаты — по `Payment.id` (groupBy/aggregate), не через include с items (join размножит). Особенно staff.collected и cashflow.inflow.
13. **Незакрытая смена.** `closingBalance=null` не считать 0 — помечать `openShiftsCount`, исключать из expectedClosing. avgLeadTime — только по completedAt, fallback startedAt||createdAt, отрицательные отбросить.
14. **Платёж без заказа при branchId.** Аванс без order при активном фильтре исключается — задокументировано (§5.2). По возможности — строка «без филиала».
15. **Отрицательные остатки.** В totalValue учитывать как есть, выделять `negativeStock` предупреждением. purchasePrice null → 0.
16. **Часовой пояс.** Сервер UTC, TJ=UTC+5 — упрощение. Границы включительные. Предупредить владельца про пограничные ночные заказы (могут попасть в соседний день/бакет).
17. **lowStock vs outOfStock.** lowStock только `qty<=minStock && minStock>0`. Товары с minStock=0 не дефицитные, но `outOfStock` (`qty<=0 && isActive`) выводить отдельно — нулевой расходник останавливает производство.
18. **Отменённый заказ и деньги.** CANCELLED исключается из billed/debt/receivables, но Payment по нему мог остаться в collected (деньги получены до отмены, возврат — отдельно). Денежные метрики от Payment, товарно-выручочные от Order — не смешивать.
